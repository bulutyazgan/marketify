- **Supabase defaults views to `security_invoker=off` (effectively DEFINER)**; it will fail the `security_definer_view` advisor. Every view we create must be followed with `alter view <name> set (security_invoker = on)` to match the project standard at §4.1 line 375. Applied to `submission_reuse_view` in US-006.
- **RLS helper pattern (use for every policy that scopes to the caller):** two SQL `stable` helpers `public.current_user_id()` (returns `nullif(auth.jwt() ->> 'sub', '')::uuid`) and `public.current_user_role()` (returns `nullif(auth.jwt() ->> 'role', '')::public.user_role`). Both use `set search_path = ''` and schema-qualify every ref. `nullif` on the empty string makes a missing claim safely return NULL (all `id = current_user_id()` checks fail closed); for `current_user_role()` this is a hardening deviation from the spec's verbatim `(auth.jwt() ->> 'role')::user_role` — intentional, keeps the fn from raising on malformed JWTs. Policies call these helpers instead of inlining `auth.jwt()` for readability + single-source-of-truth. Ref: `supabase/migrations/20260417140000_us_009_rls_policies.sql`.
- **The `submission_reuse_view` → `submission_reuse_count(uuid)` function swap (US-009 resolution):** the view with `security_invoker=on` would be filtered by the caller's `submissions_lister_read` policy and hide cross-lister reuses (the ReuseBadge's entire purpose). The fix: **drop the view** and replace with a SECURITY DEFINER function `public.submission_reuse_count(p_submission_id uuid) returns integer` that (1) does its own ownership check via join through `submissions → applications → listings.lister_id = current_user_id()`, (2) returns NULL when the caller isn't the listing owner, (3) bypasses RLS to count cross-lister reuses. Revoked from public, granted to authenticated + service_role. Keep this pattern for any other cross-tenant aggregate the UI needs.
- **Verify RLS via a transactional `begin; ... rollback;` block:** seed fresh fixtures with known `auth.uid`s, then `set local role authenticated` + `set local "request.jwt.claims" = '{"sub":"<uuid>","role":"creator"}'` and assert row visibility via `insert into _asserts ...` rows that you `select *` at the end before rolling back. Important: `grant insert, select on _asserts to authenticated` before switching roles, else you'll get `permission denied for table _asserts`. `PERFORM` is PL/pgSQL only — don't use it outside a DO block. Check `information_schema.columns` before writing fixtures so INSERTs match real schema (columns migrate between blocks — e.g. `listing_version_id` lives on `applications`, not `submissions`).
- **Trigger function hardening pattern (mandatory for every new trigger / PL/pgSQL function in this repo):**
  1. Declare `set search_path = ''` on the function (CREATE OR REPLACE drops prior SET clauses, so restate it on every redefinition).
  2. With empty `search_path`, qualify EVERY user-defined ref: tables `public.<tbl>`, types/enum casts `public.<type>` (e.g. `'tiktok'::public.platform`), `app_private.<fn>` for sibling functions.
  3. For pg_catalog **functions** (`now()`, `hashtextextended()`, `pg_advisory_xact_lock()`, etc.): bare is fine — `pg_catalog` is always implicitly searched even with empty `search_path`. Schema-qualifying as `pg_catalog.<fn>()` also works for true functions.
  4. **GREATEST / LEAST / COALESCE / NULLIF are SQL parser constructs, NOT functions.** They CANNOT be schema-qualified (`pg_catalog.greatest()` raises 42883). Always write them bare.
  5. SECURITY DEFINER functions especially require empty `search_path` — without it, a caller-controlled search_path can shadow `public.<table>` and divert writes. The advisor `function_search_path_mutable` flags this at WARN.
- **US-008 TODO — out-of-order Apify webhook delivery:** `app_private.denorm_metrics` uses `greatest(metrics_fetched_at, new.fetched_at)` to guard the timestamp column, but the surrounding column UPDATEs (e.g. `tiktok_follower_count`) are unconditional. - **US-008 nice-to-have — advisory lock collision risk:** `app_private.denorm_metrics` uses single-arg `pg_advisory_xact_lock(hashtextextended(...))`. Two distinct (social_link_id, scrape_mode) strings can collide to the same 64-bit key, causing unrelated creators to serialize unnecessarily. Not a correctness issue (only throughput). Standard fix is the two-arg form `pg_advisory_xact_lock(hashtext(social_link_id::text), hashtext(scrape_mode::text))`. - **`public.events` append-only is a social contract, not a DB-level guarantee.** US-008 reproduces the spec literally (`revoke all on public.events from anon, authenticated`), but `service_role` and `postgres` retain every grant including UPDATE/DELETE — by design per spec §4.4 lines 1083–1085 ("Service role only. No application read/write policy"). Any edge function or trigger that writes to `events` must treat INSERT as the only safe verb; there is no DB trigger that blocks UPDATE/DELETE. If stronger enforcement is ever desired, add a BEFORE UPDATE OR DELETE trigger raising `RAISE EXCEPTION` (this would also block maintenance scripts — considered too restrictive for v1).
- **In PL/pgSQL with `set search_path = ''`, `text[] || 'literal'` resolves to `anyarray || anyarray` and crashes with `22P02 malformed array literal`.** Postgres chooses the array+array overload for unknown-type literals. Always cast the right-hand side: `changed_fields || 'price_cents'::text`. Typed text columns (e.g. `new.version_bump_reason`) work bare because the operator resolves to `anyarray || anyelement` when the right side has a concrete element type. This gotcha is independent of the `public.<type>` schema-qualification pattern already in Codebase Patterns. Resolved in US-010 via `us_010_fix_array_append` follow-up migration.

- **JWT helper contract (`supabase/functions/_shared/jwt.ts`):** `signJwt({sub, role, session_id}, {ttlSeconds?})` + `verifyJwt(token): JwtClaims`. HS256 via Web Crypto. Secret from `MARKETIFY_JWT_SECRET` (NOT the Supabase project JWT secret). Claims are `{sub, role, session_id, iat, exp}` — `session_id` is spec-required per §4.4 (story AC said only 4 claims; spec wins). `verifyJwt` enforces `alg=HS256`, exp > now, iat ≤ now+60s, and validates claim shapes. Any JSON/base64/atob malformation surfaces as `"Malformed JWT"`. Uses `crypto.subtle.verify` (not re-sign + compare) with a verify-only-usage key.
- **CORS headers shared module (`supabase/functions/_shared/cors.ts`):** `corsHeaders` allows `*` origin — safe because edge functions authenticate via `Authorization: Bearer <marketify_jwt>`, not cookies. Every edge function must splat this into responses and handle an `OPTIONS` pre-flight with `{ status: 204, headers: corsHeaders }`.
- **Atomic multi-insert edge-function flow uses a PL/pgSQL RPC, not back-to-back supabase-js inserts.** Two `supabase.rpc`/`.insert()` calls are two separate PostgREST transactions, so a mid-flight crash leaves orphan rows. The established pattern (US-014) is a single `public.<flow>_rpc` PL/pgSQL function invoked via `supabase.rpc(...)` from the edge function — PL/pgSQL wraps the body in an implicit transaction. Catch `unique_violation` with `GET STACKED DIAGNOSTICS v_constraint = constraint_name` and return `{error: CODE}` jsonb; let everything else `RAISE` so the edge function can log+500. Implicit-constraint names (`<table>_<column>_key`) are Postgres defaults — ALWAYS comment the expected names in the exception block, since a future migration that renames them would silently downgrade 409s to 500s. Revoke execute from anon/authenticated/public; grant only to service_role (no SECURITY DEFINER needed — service_role already has BYPASSRLS).
- **Supabase edge-function secrets are set via `supabase secrets set --env-file <tmpfile> --project-ref <ref>`, not via Supabase MCP.** The MCP has no secrets tool. Create a temp file containing `NAME=value`, run the CLI, delete the temp file. Never echo the secret value into chat. After setting, re-invoke the edge function — secret names are available via `Deno.env.get(...)`. Running the edge function before the secret is set surfaces as a 500 at the point the helper tries to read it (for `MARKETIFY_JWT_SECRET`, that's `signJwt` → `"MARKETIFY_JWT_SECRET is not set"` → caught → returned as `{error: "JWT_ERROR"}` in the current auth-* pattern).
- **Edge-function smoke tests live under `scripts/test/<fn-name>.ts`, invoked with `bun run`.** Read `SUPABASE_URL` + `SUPABASE_ANON_KEY` from `process.env` (bun auto-loads `.env`). Send `Authorization: Bearer <anon>` + `apikey: <anon>` headers even for `verify_jwt=false` functions — the Supabase gateway still requires a valid apikey at the edge. Use `Date.now()` for unique test fixture values so repeat runs don't collide. Every conflict branch (e.g. 409 USERNAME_TAKEN vs 409 EMAIL_TAKEN) gets its own case — the constraint-name mapping in the RPC's exception handler is only validated by cases that actually hit each branch. Clean up test rows via `execute_sql` after each run.
- **Anonymous `unique` declarations in CREATE TABLE get Postgres default names (`<table>_<column>_key`)** — e.g. `username citext not null unique` on `public.users` produces `users_username_key`. Any PL/pgSQL that inspects `constraint_name` depends on this. Verify via `select conname from pg_constraint where conrelid = 'public.<table>'::regclass` before hardcoding a name. If you later want to rename, drop+re-add with explicit `constraint <name> unique (col)` — and update every dependent exception handler in the same migration.
- **Apify REST endpoint + slug encoding:** use `POST https://api.apify.com/v2/acts/<rest-slug>/runs?token=<APIFY_KEY>&waitForFinish=<secs>`. REST slugs replace `/` with `~` (`clockworks~tiktok-scraper`, `apify~instagram-scraper`). `waitForFinish` makes Apify block up to N seconds before returning the run object — if the run exceeds N, Apify returns `status=RUNNING` and webhooks pick up the terminal state later. Response shape is `{ data: { id, defaultDatasetId, status, ... } }`. For this project the wrapper at `supabase/functions/_shared/apify.ts` uses direct REST (not the `apify-client` SDK) because Deno in Edge Runtime handles plain fetch cleanly and `Deno.env.get("APIFY_KEY")` is the sanctioned way to read the token.
- **Never echo external API error bodies into thrown Error strings when the request URL carries a secret in the query string.** Apify (and several other providers) sometimes include the request URL in error payloads, so `throw new Error(\`... ${body}\`)` can land the token in edge-function logs. Throw a fixed-string error with status + slug only; if you need the body for debugging, log it separately with redaction.
- **Deno tests for `_shared/` modules live alongside them as `<name>.test.ts` and run with `deno test --allow-env supabase/functions/_shared/<name>.test.ts`.** Assertions come from `jsr:@std/assert@1`. Mock fetch by replacing `globalThis.fetch` with a spy that records `{url, init}` and returns a canned `Response`; always restore in a `try/finally`. Deno isn't on every dev machine; `brew install deno` on macOS is a one-liner. If Deno isn't available in a given iteration, document it in the progress entry and have the CI for this run the tests.
- **Public async wrappers must be declared `async` if they do any synchronous validation that might throw**, so the throw surfaces as a Promise rejection rather than a synchronous exception. `assertRejects` in `@std/assert` fails with "Function throws when expected to reject" if the function throws synchronously — a real footgun that can only be caught by actually running the test.
- **Constant-time dual-secret compare for rotation-aware webhook auth:** edge functions that authenticate third-party webhooks via a shared secret must (a) compare in constant time (XOR-accumulate over `max(len(a), len(b))` bytes, iterate the longer one), (b) check BOTH `<SECRET>` and `<SECRET>_PREVIOUS` env vars without short-circuit, and (c) return false if neither env var is set. The second point is subtle: `return timingSafeEqual(a, current) || timingSafeEqual(a, previous)` short-circuits when the first matches, which means an attacker who has recovered `current` can time-distinguish whether `previous` is configured. Correct shape is `let matchA = current ? timingSafeEqual(provided, current) : false; let matchB = previous ? timingSafeEqual(provided, previous) : false; return matchA || matchB` — both evaluated unconditionally. Reference: `supabase/functions/apify-webhook/index.ts` `verifyWebhookSecret`.
- **Third-party-webhook-receiver edge functions deploy with `verify_jwt=false`** (Supabase MCP `deploy_edge_function({verify_jwt: false})`). The provider cannot send an `Authorization: Bearer <marketify-jwt>` — authentication is via the provider's own header (e.g. `X-Apify-Webhook-Secret`). The Supabase API gateway still requires a valid project `apikey` though, so smoke tests must send `Authorization: Bearer <anon>` + `apikey: <anon>` headers in addition to the webhook-specific auth header. For Apify specifically, pair this with a `payloadTemplate` in the per-run webhook registration that injects `social_link_id` / `scrape_mode` / `run_id` / `apify_finished_at` — these fields are NOT in Apify's default event shape and are required by our receiver contract.
- **Webhook-receiver idempotency via service-role RPC + ON CONFLICT DO NOTHING:** the receiver's persistence step goes through a PL/pgSQL RPC that `INSERT ... ON CONFLICT (<run-id-col>) WHERE <run-id-col> IS NOT NULL DO NOTHING` and returns jsonb `{inserted: bool, duplicate: bool, <id>?: uuid}`. Paired with a partial unique index on the run-id column, replays of the same webhook are safe. Revoke the RPC from anon/authenticated/public and grant only to service_role. The RPC is invoked by the edge function with a service-role Supabase client — NO `SECURITY DEFINER` needed because service_role already has BYPASSRLS. Reference: `public.apify_webhook_persist_tiktok_profile` + `metric_snapshots_run_uniq` (US-006/016).
- **Scope env-var reads to the branch that actually needs them.** A single top-of-handler `Deno.env.get("OPTIONAL_KEY")` guard that 500s on missing keys will trip branches that don't need the key. For the apify-webhook, `APIFY_KEY` is only needed to fetch the dataset on the SUCCEEDED branch — gating the whole handler on it broke the FAILED/TIMED_OUT/ABORTED branches (which have no dataset). Move the lookup inside the conditional, and only return `SERVER_MISCONFIGURED` when you actually can't proceed.
- **`supabase/functions/<name>-live.ts` convention for Apify-touching e2e tests:** any smoke test that spends real Apify credits lives alongside the cheap-path smoke as `<fn>-live.ts`, is NOT part of the default test suite, and documents what it costs + what external state it touches. DB-side verification (snapshot row + denorm trigger) is done separately via MCP `execute_sql` after the script completes.
- **Apify `payloadTemplate` requires `shouldInterpolateStrings: true` for quoted `"{{placeholder}}"` substitution.** Without this flag, quoted placeholders stay literal (`"{{resource.id}}"` delivered as-is) and the receiver's validation layer 400s. Bare unquoted `{{var}}` substitutes raw JSON but breaks `JSON.stringify(payloadTemplate)` because the placeholder isn't valid JSON. Rule: every webhook spec we build has `shouldInterpolateStrings: true` and uses quoted placeholders exclusively. Reference: `supabase/functions/_shared/apify.ts` `ApifyWebhookSpec` + `supabase/functions/auth-signup-creator/index.ts`.
- **Apify transitional run statuses are READY, RUNNING, TIMING-OUT, ABORTING.** When `waitForFinish` expires before a run reaches a terminal state, Apify returns one of these four — in all four cases the run continues and the registered webhook will still fire the terminal event later. Map all four to 'refreshing' in any caller that derives a metrics-status from the immediate actor-run response; only `SUCCEEDED` should map to 'fresh' and any other (FAILED/TIMED-OUT/ABORTED) to 'failed'. Don't special-case only RUNNING.
- **Reanimated reduced-motion pattern (callers + helpers):** `withReducedMotion(config, reduced)` in `src/design/motion.ts` spreads `reduceMotion: ReduceMotion.Always` onto any Reanimated `SpringConfig`/`TimingConfig` when the caller's `useReducedMotion()` is true. Reanimated v4 collapses `Always`-flagged animations to an instant snap on `onStart` — this is the sanctioned no-op; do NOT roll your own `withTiming(target, {duration: 0})` wrapper. Call shape: `withSpring(target, withReducedMotion<WithSpringConfig>(SPRING_SOFT, reduced))` — the **explicit `<WithSpringConfig>` generic is mandatory** because the presets are declared `as const satisfies WithSpringConfig` which narrows them to readonly literal types; under `strict` TS those literals don't satisfy the helper's `C extends {reduceMotion?: ReduceMotion}` constraint without widening. Inline the helper inside `withSpring` rather than caching to a `const spring` — `reduced` can flip at runtime when the user toggles OS reduce-motion. `useReducedMotion()` in `src/design/useReducedMotion.ts` uses a module-scope `cached: boolean | null` so only the first hook mount per app session pays the async `AccessibilityInfo.isReduceMotionEnabled()` race — subsequent mounts read the cached value synchronously and never flash `false` on devices with reduce-motion on.
- **Animatable iOS shadow props in `useAnimatedStyle`: `shadowOpacity` + `shadowRadius` only — NOT `shadowOffset`.** `shadowOffset` is a compound `{width, height}` object prop that Reanimated 4 does not interpolate through `useAnimatedStyle`; the underlying UIKit shadow stays at its `StyleSheet` value and the offset never moves. For the neubrutalist "shadow collapses to (0,0) on press" spec (docs/design.md §1.5 line 118), the sanctioned pattern is to **animate `shadowOpacity` from 1 → 0** while simultaneously translating the view `(0,0) → (3,3)` — the shadow visually disappears into the pressed-down button. Reference: `src/components/primitives/Button.tsx`.
- **Neubrutalist Pressable pattern (`src/components/primitives/Button.tsx`):** one shared value `pressed: 0 → 1` drives `translateX/Y`, `scale`, and `shadowOpacity` in a single `useAnimatedStyle`. `onPressIn` fires haptic + springs to 1, `onPressOut` springs to 0. Guard BOTH handlers with `if (!interactive) return;` (the disabled-state `styles.disabled` clears shadowOpacity, but `handlePressOut` still dispatches a spring if unguarded — stale-closure hygiene). `interactive = !disabled && !loading` — loading implies the press is blocked too. Pressable's own `disabled={!interactive}` prevents touch events from reaching RN when disabled, but the guards are defense-in-depth. Every primitive that needs a press animation (CampaignCard, Chip, BottomSheet drag handle) should reuse this shape.
- **Toggle-chip accessibility: `accessibilityRole="checkbox"` with `accessibilityState={{ checked }}`, NOT `"button"` with `{ selected }`.** `role="button"` ignores `aria-selected` — most screen readers silently drop it, so a filter chip reads as an inert button and the user can't tell what's on vs off. `checkbox` maps cleanly to `aria-checked` on iOS/Android and the `checked` boolean surfaces as "checked"/"not checked" in VoiceOver/TalkBack. Applies to any toggleable pill (filter chips, segmented controls, multi-select tags). Reference: `src/components/primitives/Chip.tsx`. For read-only `View`s (e.g. StatusPill), do NOT set `accessibilityRole="text"` — that role is for `Text` nodes only and iOS silently rejects it on a `View`; instead use `accessible` + `accessibilityLabel` on the `View` and let the child `Text` inherit silently.
- **`textStyles.micro` is the single source of truth for pill/chip labels** — Rubik_700Bold (stand-in for Satoshi 700 per the US-023 fallback chain), 11pt, lineHeight 14, `letterSpacing: 0.88`, `textTransform: 'uppercase'`. The 0.88 is the absolute-pt conversion of §1.3's `0.08em at 11pt` (RN letterSpacing is absolute points, not em). Every pill/chip/status-badge should apply `textStyles.micro` plus its semantic color inline, plus `allowFontScaling={false}` per the §8 checklist rule that UI chrome opts out of Dynamic Type. Added in US-026 (the first story that needed it); US-023 intentionally deferred micro since its AC enumerated only 6 keys.
- **Neubrutalist "lifted" tier is universal** — cards, buttons, chips, inputs, AND status pills all get `shadows.hard` from `src/design/tokens.ts` (offset 3,3, solid ink, opacity 1, radius 0). The spec's "flat" tier (§1.5 line 119) is reserved for content inside cards (e.g. EmptyState illustrations, dividers) — not standalone surfaces. Interactive lifted surfaces collapse the shadow on press via the animated `shadowOpacity` pattern; read-only lifted surfaces (StatusPill) just spread `shadows.hard` into their static style. Don't skip the shadow on small pills "because they're small" — it's what makes the status pills visually readable at a glance in a dense inbox row.
- **`useAnimatedStyle` must NOT close over JS state (`useState`) when that state is updated from a component event** (e.g. `onLayout` → `setState`). Reanimated serializes the worklet closure at render time: between a `setState` write and the subsequent re-render, the worklet keeps reading the stale prior value, and on first mount the worklet captures the initial `0` before the first layout re-render lands — producing a one-frame flicker. **The pattern for layout-derived animation inputs is a dedicated `useSharedValue` written from the event handler directly.** In `SkeletonCard`, `onLayout` writes to `width.value` (shared value) rather than `setWidth` (state), the worklet reads `width.value` synchronously, and `bandWidth` is derived inline inside the worklet. Same rule applies to any future primitive that animates based on its own measured size (e.g. BottomSheet snap points, Toast progress bars). Ref: `src/components/primitives/SkeletonCard.tsx`.
- **Non-interactive Pressables should set `accessible={false}`** (not `accessible` defaulted to true with a missing `accessibilityRole`). When a card has no `onPress`, VoiceOver/TalkBack would otherwise land on an unlabeled, roleless wrapper before descending into the children — confusing for users and technically a spec violation (nodes reporting as focusable must expose a role). Pattern: `accessible={interactive}` on the wrapping Pressable, `accessibilityRole` gated to interactive only, `accessibilityLabel` gated to interactive only. Children (title, handle text) then become the actual focus targets and read naturally. Applies to every touchable primitive that can be rendered in a read-only context (CampaignCard, future ProfileCard, etc.).
- **RN `Modal` opens a new window that does not inherit the root `GestureHandlerRootView` on Android — always nest a fresh `GestureHandlerRootView` inside the Modal body** for any gesture-driven content. iOS happens to work without this because the gesture handler registers globally, but Android silently drops pan/fling events. The sanctioned pattern (BottomSheet) is `Modal(transparent + statusBarTranslucent) → GestureHandlerRootView → backdrop Animated.View + GestureDetector(sheet)`. `statusBarTranslucent` is also required on Android so the backdrop scrim covers the status bar area — without it the top strip stays at the app background color during the slide-up. Ref: `src/components/primitives/BottomSheet.tsx`.
- **Spread-order of `shadows.hard` silently clobbers component-local shadow overrides.** `shadows.hard` is a full shadow tuple (offset 3,3, color ink, opacity 1, radius 0) and spreading it AFTER a local `styles.sheet` that tries to override `shadowOffset` will replace the local value. Postmortem from US-028: a bottom-pinned sheet attempted `shadowOffset: {width: 0, height: -3}` in `styles.sheet` + `shadows.hard` spread later in the JSX array — the upward offset was silently overridden back to (3,3). Rule: either accept the universal (3,3) hard-shadow (see the "lifted tier is universal" pattern) or write the full shadow tuple inline without the token. Don't layer partial overrides on top of the token.
- **RN `Modal` portal + `accessibilityViewIsModal`:** for modal presentations (bottom sheets, full-screen dialogs), set `accessibilityViewIsModal` on the foreground Animated.View so screen readers trap focus inside it and skip the parent view hierarchy. Pair with a focusable dismiss control (a `Pressable` backdrop is fine) but mark the backdrop `Pressable` as `accessible={false}` + `importantForAccessibility="no"` so focus doesn't land on an unlabeled "Dismiss sheet" wrapper — rely on the modal's own close affordance (title/CTA) as the primary focus target. Backdrop `Animated.View` should carry `pointerEvents="box-none"` so its children (the Pressable) receive taps while the animated parent itself doesn't intercept events inside the sheet area.
- **Imperative toast API via React context (not a global singleton).** `ToastProvider` mounts once near the root (`app/_layout.tsx`) and stores a single `toast | null` slot in state; `useToast().show({ message, variant, duration })` replaces the current slot if one exists (cancel the pending setTimeout first), auto-dismisses after `duration ?? 2500` ms, and clears the slot via a `setToast((cur) => cur?.id === id ? null : cur)` functional update so a replacement after the timer fires doesn't bounce. `useToast()` throws outside the provider — catching the error at dev-time is better than a silent no-op. Exit is a single-slot banner (NOT a queue), consistent with docs/design.md §6's singular `show(message, variant, duration)` signature. Ref: `src/components/primitives/Toast.tsx`.
- **Progress-bar countdown in Reanimated uses `withTiming(0, {duration, easing: Easing.linear})` on a 0→1 shared value, and the track width lives in a second shared value written from `onLayout`.** Same anti-stale-capture rule from the SkeletonCard pattern applies — never read `useState` width inside the worklet. On toast-id change, reset `progress.value = 1` synchronously before starting the new `withTiming`, and `cancelAnimation(progress)` on unmount / next-toast. Under reduced motion, skip the timing entirely (leave progress at 1) — never gimmick a zero-duration timing to simulate the snap; let the `withReducedMotion` helper do it, or just omit the animation.
- **`accessibilityLiveRegion` is Android-only and redundant with `accessibilityRole="alert"` for transient banners.** On iOS, `LiveRegion` is ignored and `role=alert` is what triggers VoiceOver announcement; on Android, `role=alert` already implies a polite live region per the platform's AccessibilityNodeInfo semantics. Setting both just adds noise. For toast banners, use `accessibilityRole="alert"` only. If you specifically need an "assertive" live region on Android (rare — reserve for security/error states that preempt current speech), then set `accessibilityLiveRegion="assertive"`.
- **Expo Router v6 routing rules (US-029):** (a) files/directories prefixed with `_` in `app/` are SILENTLY EXCLUDED from routing — don't use `app/_dev/primitives.tsx` expecting a `/_dev/primitives` URL, it won't register and `router.navigate` will 404 with no warning. (b) Route groups in `(parens)` are valid everywhere and the parens are stripped from the URL (`app/(dev)/primitives.tsx` → URL `/primitives`). (c) `<Redirect href>` accepts either the full filesystem path `/(group)/segment` OR the URL-stripped `/segment` — expo-router resolves both against the route map. (d) There is NO native build-mode gate to conditionally register a route; the sanctioned "dev-only route" pattern is runtime: `if (!__DEV__) return <Redirect href="/" />;` inside the route component. (e) If a story's URL AC conflicts with these rules (e.g. wants a literal `_` in a URL path), document as a spec gap and use the closest convention-compatible path. Dev-only catalog lives at `app/(dev)/primitives.tsx` + `src/screens/_dev/PrimitivesPreview.tsx` as the canonical reference.
- **`react-native-mmkv` v4 is a Nitro module and will NOT load in Expo Go** — the `createMMKV()` call throws `NitroModules are not supported in Expo Go` at module-import time, which cascades to "Route './_layout.tsx' is missing the required default export" and a hard app-boot failure. Every iteration still targets Expo Go (dev-client build planned in US-031), so persist credentials with `expo-secure-store` instead — it ships in the Expo SDK, is iOS Keychain / Android Keystore backed, and works in Expo Go natively. When the dev client eventually lands, `src/lib/storage.ts` is the single swap point if the team wants MMKV back. Note MMKV v4 also renamed `.delete(key)` → `.remove(key)` from v2; pattern is documented even if we don't use it right now.
- **SecureStore is async; a sync read path needs a module-scope in-memory cache.** The Supabase `global.fetch` wrapper needs a synchronous JWT read per request. Pattern: keep `cachedToken: string | null` at module scope in `src/lib/storage.ts`, expose `getCachedToken()` for sync callers, and have `persistToken()` / `clearPersistedAuth()` update the cache BEFORE awaiting the SecureStore write so the very-next fetch sees the new value. AuthProvider hydrates the cache once on mount via `hydrateTokenFromStorage()` and gates children on a `hydrated` boolean — returns `null` until the initial read settles to avoid a flash of logged-out state.
- **Supabase typed client for custom-JWT auth (`src/lib/supabase.ts`):** `createClient<Database>(url, anonKey, { auth: { persistSession: false, autoRefreshToken: false, detectSessionInUrl: false }, global: { fetch: marketifyFetch } })`. The three `auth.*: false` flags are required — otherwise Supabase-js tries to manage its own session storage and fights the custom `Authorization` header. The fetch wrapper builds headers via `new Headers(init?.headers)` which preserves Supabase-js's internally-set `apikey` header (required by the Supabase API gateway even for authed calls), then overwrites only `Authorization` when a token is cached. Unauthed reads work because the wrapper short-circuits to plain `fetch(input, init)` when no token is cached and Supabase-js's default `Authorization: Bearer <anon>` survives.
- **`app.config.ts` extends `app.json` dynamically** — `export default ({ config }: ConfigContext): ExpoConfig => ({ ...(config as ExpoConfig), extra: { ...config.extra, SUPABASE_URL: process.env.SUPABASE_URL, ... } })`. Expo CLI auto-loads `.env` into `process.env` when the dev server starts (no `dotenv` import needed). Values surface at runtime via `Constants.expoConfig?.extra`. Dynamic `app.config.ts` + static `app.json` coexist — Expo reads both and the dynamic file gets the merged config as its `config` argument. Required for the Supabase client wiring in US-030; future env surfaces (Supabase service-role keys, feature flags, etc.) add another entry under `extra`.
- **Typed auth context pattern (`src/lib/auth.tsx`):** `AuthUser = Database['public']['Tables']['users']['Row']` and `AuthRole = Database['public']['Enums']['user_role']` derive directly from generated Supabase types — no hand-maintained shapes. `useAuth()` returns `{ user, role: user?.role ?? null, signIn(token, user), signOut() }`. Validate shape when re-hydrating from storage (a structural `isAuthUserShape` guard, not just `JSON.parse as AuthUser`) so a stale/corrupted entry from a previous schema clears itself instead of propagating `undefined` fields into React state. `signIn`/`signOut` are fire-and-forget: they `setUser(...)` synchronously and `void persistToken(...)` in the background — the in-memory cache in storage.ts is updated synchronously inside `persistToken` so the next Supabase fetch sees the new JWT immediately. Optimistic-UI flow; if the app crashes between `setUser` and the SecureStore write settling, the user just signs in again on next launch.
