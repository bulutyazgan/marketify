- **RLS helper pattern (use for every policy that scopes to the caller):** two SQL `stable` helpers `public.current_user_id()` (returns `nullif(auth.jwt() ->> 'sub', '')::uuid`) and `public.current_user_role()` (returns `nullif(auth.jwt() ->> 'role', '')::public.user_role`). - **The `submission_reuse_view` â†’ `submission_reuse_count(uuid)` function swap (US-009 resolution):** the view with `security_invoker=on` would be filtered by the caller's `submissions_lister_read` policy and hide cross-lister reuses (the ReuseBadge's entire purpose). - **Verify RLS via a transactional `begin; ... rollback;` block:** seed fresh fixtures with known `auth.uid`s, then `set local role authenticated` + `set local "request.jwt.claims" = '{"sub":"<uuid>","role":"creator"}'` and assert row visibility via `insert into _asserts ...` rows that you `select *` at the end before rolling back. Important: `grant insert, select on _asserts to authenticated` before switching roles, else you'll get `permission denied for table _asserts`. `PERFORM` is PL/pgSQL only â€” don't use it outside a DO block. Check `information_schema.columns` before writing fixtures so INSERTs match real schema (columns migrate between blocks â€” e.g. `listing_version_id` lives on `applications`, not `submissions`).
- **Trigger function hardening pattern (mandatory for every new trigger / PL/pgSQL function in this repo):**
  1. Declare `set search_path = ''` on the function (CREATE OR REPLACE drops prior SET clauses, so restate it on every redefinition).
  2. With empty `search_path`, qualify EVERY user-defined ref: tables `public.<tbl>`, types/enum casts `public.<type>` (e.g. `'tiktok'::public.platform`), `app_private.<fn>` for sibling functions.
  3. For pg_catalog **functions** (`now()`, `hashtextextended()`, `pg_advisory_xact_lock()`, etc.): bare is fine â€” `pg_catalog` is always implicitly searched even with empty `search_path`. Schema-qualifying as `pg_catalog.<fn>()` also works for true functions.
  4. **GREATEST / LEAST / COALESCE / NULLIF are SQL parser constructs, NOT functions.** They CANNOT be schema-qualified (`pg_catalog.greatest()` raises 42883). Always write them bare.
  5. SECURITY DEFINER functions especially require empty `search_path` â€” without it, a caller-controlled search_path can shadow `public.<table>` and divert writes. The advisor `function_search_path_mutable` flags this at WARN.
- **US-008 TODO â€” out-of-order Apify webhook delivery:** `app_private.denorm_metrics` uses `greatest(metrics_fetched_at, new.fetched_at)` to guard the timestamp column, but the surrounding column UPDATEs (e.g. `tiktok_follower_count`) are unconditional. - **US-008 nice-to-have â€” advisory lock collision risk:** `app_private.denorm_metrics` uses single-arg `pg_advisory_xact_lock(hashtextextended(...))`. - **`public.events` append-only is a social contract, not a DB-level guarantee.** US-008 reproduces the spec literally (`revoke all on public.events from anon, authenticated`), but `service_role` and `postgres` retain every grant including UPDATE/DELETE â€” by design per spec Â§4.4 lines 1083â€“1085 ("Service role only. - **In PL/pgSQL with `set search_path = ''`, `text[] || 'literal'` resolves to `anyarray || anyarray` and crashes with `22P02 malformed array literal`.** Postgres chooses the array+array overload for unknown-type literals. Typed text columns (e.g. `new.version_bump_reason`) work bare because the operator resolves to `anyarray || anyelement` when the right side has a concrete element type. - **US-010 hard dependency for US-038 and US-049-055:** the cascade trigger (`app_private.bump_listing_version`) swaps `listings.current_version_id` to the new version, but it does NOT refresh the cached `listings.min_followers_tiktok` / `min_followers_instagram` threshold columns. - **`pg_net.net.http_post(url := NULL, ...)` is a silent no-op, not an error.** When the URL or any required string is NULL (e.g. a `vault.decrypted_secrets` lookup misses), pg_net returns a request id without raising and `cron.job_run_details` logs the run as `succeeded`. - **`pg_cron` job-name idempotency requires a guarded `cron.unschedule`.** `cron.schedule` raises on duplicate jobname; `cron.unschedule(text)` raises when the name is unknown.
