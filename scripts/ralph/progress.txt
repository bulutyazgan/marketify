- **Supabase defaults views to `security_invoker=off` (effectively DEFINER)**; it will fail the `security_definer_view` advisor. - **RLS helper pattern (use for every policy that scopes to the caller):** two SQL `stable` helpers `public.current_user_id()` (returns `nullif(auth.jwt() ->> 'sub', '')::uuid`) and `public.current_user_role()` (returns `nullif(auth.jwt() ->> 'app_role', '')::public.user_role`). Both use `set search_path = ''` and schema-qualify every ref. `nullif` on the empty string makes a missing claim safely return NULL (all `id = current_user_id()` checks fail closed); for `current_user_role()` this is a hardening deviation from the spec's verbatim `(auth.jwt() ->> 'role')::user_role` — intentional, keeps the fn from raising on malformed JWTs. **Note the `app_role` claim (not `role`)** — the bare `role` claim is reserved by PostgREST for the Postgres DB role (always `"authenticated"` on our tokens); see Pattern #JWT-Claims below for why. Policies call these helpers instead of inlining `auth.jwt()` for readability + single-source-of-truth. Ref: `supabase/migrations/20260418020000_us_jwt_app_role_claim.sql` (supersedes the `role`-claim version from `us_009_rls_policies.sql`).
- **The `submission_reuse_view` → `submission_reuse_count(uuid)` function swap (US-009 resolution):** the view with `security_invoker=on` would be filtered by the caller's `submissions_lister_read` policy and hide cross-lister reuses (the ReuseBadge's entire purpose). The fix: **drop the view** and replace with a SECURITY DEFINER function `public.submission_reuse_count(p_submission_id uuid) returns integer` that (1) does its own ownership check via join through `submissions → applications → listings.lister_id = current_user_id()`, (2) returns NULL when the caller isn't the listing owner, (3) bypasses RLS to count cross-lister reuses. Revoked from public, granted to authenticated + service_role. Keep this pattern for any other cross-tenant aggregate the UI needs.
- **Verify RLS via a transactional `begin; ... rollback;` block:** seed fresh fixtures with known `auth.uid`s, then `set local role authenticated` + `set local "request.jwt.claims" = '{"sub":"<uuid>","role":"creator"}'` and assert row visibility via `insert into _asserts ...` rows that you `select *` at the end before rolling back. Important: `grant insert, select on _asserts to authenticated` before switching roles, else you'll get `permission denied for table _asserts`. `PERFORM` is PL/pgSQL only — don't use it outside a DO block. Check `information_schema.columns` before writing fixtures so INSERTs match real schema (columns migrate between blocks — e.g. `listing_version_id` lives on `applications`, not `submissions`).
- **Trigger function hardening pattern (mandatory for every new trigger / PL/pgSQL function in this repo):**
  1. Declare `set search_path = ''` on the function (CREATE OR REPLACE drops prior SET clauses, so restate it on every redefinition).
  2. With empty `search_path`, qualify EVERY user-defined ref: tables `public.<tbl>`, types/enum casts `public.<type>` (e.g. `'tiktok'::public.platform`), `app_private.<fn>` for sibling functions.
  3. For pg_catalog **functions** (`now()`, `hashtextextended()`, `pg_advisory_xact_lock()`, etc.): bare is fine — `pg_catalog` is always implicitly searched even with empty `search_path`. Schema-qualifying as `pg_catalog.<fn>()` also works for true functions.
  4. **GREATEST / LEAST / COALESCE / NULLIF are SQL parser constructs, NOT functions.** They CANNOT be schema-qualified (`pg_catalog.greatest()` raises 42883). Always write them bare.
  5. SECURITY DEFINER functions especially require empty `search_path` — without it, a caller-controlled search_path can shadow `public.<table>` and divert writes. The advisor `function_search_path_mutable` flags this at WARN.
- **US-008 TODO — out-of-order Apify webhook delivery:** `app_private.denorm_metrics` uses `greatest(metrics_fetched_at, new.fetched_at)` to guard the timestamp column, but the surrounding column UPDATEs (e.g. `tiktok_follower_count`) are unconditional. If a webhook with an older `fetched_at` lands after a newer one (different `apify_run_id`, so the dedup index doesn't catch it), the trigger will overwrite fresh denorm columns with stale values even though the timestamp itself refuses to roll back. The spec §15 has the same gap. - **US-008 nice-to-have — advisory lock collision risk:** `app_private.denorm_metrics` uses single-arg `pg_advisory_xact_lock(hashtextextended(...))`. Two distinct (social_link_id, scrape_mode) strings can collide to the same 64-bit key, causing unrelated creators to serialize unnecessarily. Not a correctness issue (only throughput). Standard fix is the two-arg form `pg_advisory_xact_lock(hashtext(social_link_id::text), hashtext(scrape_mode::text))`. - **`public.events` append-only is a social contract, not a DB-level guarantee.** US-008 reproduces the spec literally (`revoke all on public.events from anon, authenticated`), but `service_role` and `postgres` retain every grant including UPDATE/DELETE — by design per spec §4.4 lines 1083–1085 ("Service role only. No application read/write policy"). Any edge function or trigger that writes to `events` must treat INSERT as the only safe verb; there is no DB trigger that blocks UPDATE/DELETE. If stronger enforcement is ever desired, add a BEFORE UPDATE OR DELETE trigger raising `RAISE EXCEPTION` (this would also block maintenance scripts — considered too restrictive for v1).
- **In PL/pgSQL with `set search_path = ''`, `text[] || 'literal'` resolves to `anyarray || anyarray` and crashes with `22P02 malformed array literal`.** Postgres chooses the array+array overload for unknown-type literals. Always cast the right-hand side: `changed_fields || 'price_cents'::text`. Typed text columns (e.g. `new.version_bump_reason`) work bare because the operator resolves to `anyarray || anyelement` when the right side has a concrete element type. This gotcha is independent of the `public.<type>` schema-qualification pattern already in Codebase Patterns. Resolved in US-010 via `us_010_fix_array_append` follow-up migration.

- **Edge functions are Deno, not RN — exclude `supabase/` from the Expo toolchain.** tsc (`tsconfig.json` exclude) and eslint (`eslint.config.js` ignores) both skip `supabase/**`. Edge-function code imports `Deno.env`, uses Web Crypto, and runs under Supabase Edge Runtime. - **JWT helper contract (`supabase/functions/_shared/jwt.ts`):** `signJwt({sub, app_role, session_id}, {ttlSeconds?})` + `verifyJwt(token): JwtClaims`. HS256 via Web Crypto. Secret from `MARKETIFY_JWT_SECRET` (NOT the Supabase project JWT secret). Claims are `{sub, role: "authenticated", app_role: "creator"|"lister", session_id, iat, exp}`. `session_id` is spec-required per §4.4 (story AC said only 4 claims; spec wins). **`role` is ALWAYS `"authenticated"`** — it names the Postgres DB role PostgREST switches to. The app-level creator/lister distinction is on a separate `app_role` claim. `verifyJwt` enforces `alg=HS256`, `role === "authenticated"`, `app_role ∈ {creator, lister}`, exp > now, iat ≤ now+60s, and validates claim shapes. Any JSON/base64/atob malformation surfaces as `"Malformed JWT"`. Uses `crypto.subtle.verify` (not re-sign + compare) with a verify-only-usage key. Edge-function gate checks read `claims.app_role !== "creator"` (or `"lister"`), never `claims.role`. RLS helpers read `auth.jwt() ->> 'app_role'` (see RLS helper pattern above).
- **CORS headers shared module (`supabase/functions/_shared/cors.ts`):** `corsHeaders` allows `*` origin — safe because edge functions authenticate via `Authorization: Bearer <marketify_jwt>`, not cookies. Every edge function must splat this into responses and handle an `OPTIONS` pre-flight with `{ status: 204, headers: corsHeaders }`.
- **Atomic multi-insert edge-function flow uses a PL/pgSQL RPC, not back-to-back supabase-js inserts.** Two `supabase.rpc`/`.insert()` calls are two separate PostgREST transactions, so a mid-flight crash leaves orphan rows. The established pattern (US-014) is a single `public.<flow>_rpc` PL/pgSQL function invoked via `supabase.rpc(...)` from the edge function — PL/pgSQL wraps the body in an implicit transaction. Catch `unique_violation` with `GET STACKED DIAGNOSTICS v_constraint = constraint_name` and return `{error: CODE}` jsonb; let everything else `RAISE` so the edge function can log+500. Implicit-constraint names (`<table>_<column>_key`) are Postgres defaults — ALWAYS comment the expected names in the exception block, since a future migration that renames them would silently downgrade 409s to 500s. Revoke execute from anon/authenticated/public; grant only to service_role (no SECURITY DEFINER needed — service_role already has BYPASSRLS).
- **Supabase edge-function secrets are set via `supabase secrets set --env-file <tmpfile> --project-ref <ref>`, not via Supabase MCP.** The MCP has no secrets tool. Create a temp file containing `NAME=value`, run the CLI, delete the temp file. Never echo the secret value into chat. After setting, re-invoke the edge function — secret names are available via `Deno.env.get(...)`. Running the edge function before the secret is set surfaces as a 500 at the point the helper tries to read it (for `MARKETIFY_JWT_SECRET`, that's `signJwt` → `"MARKETIFY_JWT_SECRET is not set"` → caught → returned as `{error: "JWT_ERROR"}` in the current auth-* pattern).
- **Edge-function smoke tests live under `scripts/test/<fn-name>.ts`, invoked with `bun run`.** Read `SUPABASE_URL` + `SUPABASE_ANON_KEY` from `process.env` (bun auto-loads `.env`). Send `Authorization: Bearer <anon>` + `apikey: <anon>` headers even for `verify_jwt=false` functions — the Supabase gateway still requires a valid apikey at the edge. Use `Date.now()` for unique test fixture values so repeat runs don't collide. Every conflict branch (e.g. 409 USERNAME_TAKEN vs 409 EMAIL_TAKEN) gets its own case — the constraint-name mapping in the RPC's exception handler is only validated by cases that actually hit each branch. Clean up test rows via `execute_sql` after each run.
- **Anonymous `unique` declarations in CREATE TABLE get Postgres default names (`<table>_<column>_key`)** — e.g. `username citext not null unique` on `public.users` produces `users_username_key`. Any PL/pgSQL that inspects `constraint_name` depends on this. Verify via `select conname from pg_constraint where conrelid = 'public.<table>'::regclass` before hardcoding a name. If you later want to rename, drop+re-add with explicit `constraint <name> unique (col)` — and update every dependent exception handler in the same migration.
- **Apify REST endpoint + slug encoding:** use `POST https://api.apify.com/v2/acts/<rest-slug>/runs?token=<APIFY_KEY>&waitForFinish=<secs>`. REST slugs replace `/` with `~` (`clockworks~tiktok-scraper`, `apify~instagram-scraper`). `waitForFinish` makes Apify block up to N seconds before returning the run object — if the run exceeds N, Apify returns `status=RUNNING` and webhooks pick up the terminal state later. Response shape is `{ data: { id, defaultDatasetId, status, ... } }`. For this project the wrapper at `supabase/functions/_shared/apify.ts` uses direct REST (not the `apify-client` SDK) because Deno in Edge Runtime handles plain fetch cleanly and `Deno.env.get("APIFY_KEY")` is the sanctioned way to read the token.
- **Never echo external API error bodies into thrown Error strings when the request URL carries a secret in the query string.** Apify (and several other providers) sometimes include the request URL in error payloads, so `throw new Error(\`... ${body}\`)` can land the token in edge-function logs. Throw a fixed-string error with status + slug only; if you need the body for debugging, log it separately with redaction.
- **Deno tests for `_shared/` modules live alongside them as `<name>.test.ts` and run with `deno test --allow-env supabase/functions/_shared/<name>.test.ts`.** Assertions come from `jsr:@std/assert@1`. Mock fetch by replacing `globalThis.fetch` with a spy that records `{url, init}` and returns a canned `Response`; always restore in a `try/finally`. Deno isn't on every dev machine; `brew install deno` on macOS is a one-liner. If Deno isn't available in a given iteration, document it in the progress entry and have the CI for this run the tests.
- **Public async wrappers must be declared `async` if they do any synchronous validation that might throw**, so the throw surfaces as a Promise rejection rather than a synchronous exception. `assertRejects` in `@std/assert` fails with "Function throws when expected to reject" if the function throws synchronously — a real footgun that can only be caught by actually running the test.
- **Constant-time dual-secret compare for rotation-aware webhook auth:** edge functions that authenticate third-party webhooks via a shared secret must (a) compare in constant time (XOR-accumulate over `max(len(a), len(b))` bytes, iterate the longer one), (b) check BOTH `<SECRET>` and `<SECRET>_PREVIOUS` env vars without short-circuit, and (c) return false if neither env var is set. The second point is subtle: `return timingSafeEqual(a, current) || timingSafeEqual(a, previous)` short-circuits when the first matches, which means an attacker who has recovered `current` can time-distinguish whether `previous` is configured. Correct shape is `let matchA = current ? timingSafeEqual(provided, current) : false; let matchB = previous ? timingSafeEqual(provided, previous) : false; return matchA || matchB` — both evaluated unconditionally. Reference: `supabase/functions/apify-webhook/index.ts` `verifyWebhookSecret`.
- **Third-party-webhook-receiver edge functions deploy with `verify_jwt=false`** (Supabase MCP `deploy_edge_function({verify_jwt: false})`). The provider cannot send an `Authorization: Bearer <marketify-jwt>` — authentication is via the provider's own header (e.g. `X-Apify-Webhook-Secret`). The Supabase API gateway still requires a valid project `apikey` though, so smoke tests must send `Authorization: Bearer <anon>` + `apikey: <anon>` headers in addition to the webhook-specific auth header. For Apify specifically, pair this with a `payloadTemplate` in the per-run webhook registration that injects `social_link_id` / `scrape_mode` / `run_id` / `apify_finished_at` — these fields are NOT in Apify's default event shape and are required by our receiver contract.
- **Webhook-receiver idempotency via service-role RPC + ON CONFLICT DO NOTHING:** the receiver's persistence step goes through a PL/pgSQL RPC that `INSERT ... ON CONFLICT (<run-id-col>) WHERE <run-id-col> IS NOT NULL DO NOTHING` and returns jsonb `{inserted: bool, duplicate: bool, <id>?: uuid}`. Paired with a partial unique index on the run-id column, replays of the same webhook are safe. Revoke the RPC from anon/authenticated/public and grant only to service_role. The RPC is invoked by the edge function with a service-role Supabase client — NO `SECURITY DEFINER` needed because service_role already has BYPASSRLS. Reference: `public.apify_webhook_persist_tiktok_profile` + `metric_snapshots_run_uniq` (US-006/016).
- **Scope env-var reads to the branch that actually needs them.** A single top-of-handler `Deno.env.get("OPTIONAL_KEY")` guard that 500s on missing keys will trip branches that don't need the key. For the apify-webhook, `APIFY_KEY` is only needed to fetch the dataset on the SUCCEEDED branch — gating the whole handler on it broke the FAILED/TIMED_OUT/ABORTED branches (which have no dataset). Move the lookup inside the conditional, and only return `SERVER_MISCONFIGURED` when you actually can't proceed.
- **`supabase/functions/<name>-live.ts` convention for Apify-touching e2e tests:** any smoke test that spends real Apify credits lives alongside the cheap-path smoke as `<fn>-live.ts`, is NOT part of the default test suite, and documents what it costs + what external state it touches. US-016 used `clockworks~tiktok-scraper` against `@tiktok` (cheap + stable + public) with seed `social_link_id = 11111111-1111-1111-1111-111111111020` — good reference fixture for future Apify-touching stories. DB-side verification (snapshot row + denorm trigger) is done separately via MCP `execute_sql` after the script completes.
- **Apify `payloadTemplate` requires `shouldInterpolateStrings: true` for quoted `"{{placeholder}}"` substitution.** Without this flag, quoted placeholders stay literal (`"{{resource.id}}"` delivered as-is) and the receiver's validation layer 400s. Bare unquoted `{{var}}` substitutes raw JSON but breaks `JSON.stringify(payloadTemplate)` because the placeholder isn't valid JSON. Rule: every webhook spec we build has `shouldInterpolateStrings: true` and uses quoted placeholders exclusively. Reference: `supabase/functions/_shared/apify.ts` `ApifyWebhookSpec` + `supabase/functions/auth-signup-creator/index.ts`.
- **Apify transitional run statuses are READY, RUNNING, TIMING-OUT, ABORTING.** When `waitForFinish` expires before a run reaches a terminal state, Apify returns one of these four — in all four cases the run continues and the registered webhook will still fire the terminal event later. Map all four to 'refreshing' in any caller that derives a metrics-status from the immediate actor-run response; only `SUCCEEDED` should map to 'fresh' and any other (FAILED/TIMED-OUT/ABORTED) to 'failed'. Don't special-case only RUNNING.
