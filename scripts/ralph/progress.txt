## Codebase Patterns
*Append-only consolidated learnings. Each future iteration reads this BEFORE doing anything.*

- **Authoritative specs are `docs/product-plan.md`, `docs/design.md`, `docs/tech-architecture.md`.** When the story is ambiguous, read the relevant section there. Do not invent product behavior.
- **Use Supabase MCP, not the supabase CLI**, for migrations, queries, edge functions, and type generation. Migrations are append-only and named `us_NNN_description`.
- **Use mobile-mcp for any UI verification.** Never claim a UI story is done without a screenshot from `mobile_take_screenshot`.
- **Use the `feature-dev:code-reviewer` subagent before every commit.** It catches drift across iterations that you can't see in your fresh context.
- **Delegate exploration.** If a story would have you read >3 files, spawn the `Explore` subagent first and work from its summary.
- **Always regenerate `src/types/supabase.ts`** after any schema migration. Commit the regenerated file in the same commit as the migration.
- **Use `bun`, not npm/yarn/pnpm.** It is what's installed locally.
- **JWT secret is `MARKETIFY_JWT_SECRET`**, separate from Supabase's built-in JWT secret. We sign our own per docs/product-plan.md §2.
- **Apify key is `APIFY_KEY`** in `.env`. Read via `process.env.APIFY_KEY` or `Deno.env.get("APIFY_KEY")`. Never paste into source.
- **Edge functions live in `supabase/functions/<name>/index.ts`** with shared utils under `supabase/functions/_shared/`.
- **Component organization:** primitives in `src/components/primitives/`, screen-specific in `src/screens/<screen>/`, shared composites in `src/components/shared/`.
- **No emoji in product UI** — Lucide icons only (per docs/design.md).
- **One role per account in v1** — no dual-role logic.
- **Run `get_advisors({type:'security'})` after every RLS or schema change.** Address every error-level finding before commit.
- **`expo lint` default glob is `['src', 'app', 'components']`.** Empty subdirs with only `.gitkeep` cause eslint to exit 2 ("all files ignored"). Keep at least one lintable `.ts`/`.tsx` file in any of those roots (`src/index.ts` with `export {};` was used for US-001).
- **Expo bootstrap workaround:** `create-expo-app .` refuses to run when the target dir has dotfiles/dirs it considers hostile. Bootstrap into a sibling folder (e.g. `.expo-init`), then move the generated files up. Preserves `.claude/`, `.env`, `docs/`, `scripts/`.
- **Reanimated 4.x babel plugin is `react-native-worklets/plugin`** (not the legacy `react-native-reanimated/plugin`). It lives in the `react-native-worklets` package — pin it explicitly via `bunx expo install react-native-worklets` so `bun install --production` won't drop it. Included by default in fresh Expo SDK 54 templates.
- **Prefer `bunx expo install <pkg>`** over `bun add` for native modules — it pins the SDK-compatible version. Expo's install runs `bun add` under the hood when a `bun.lock` is present.
- **App entry is `"main": "expo-router/entry"`** in package.json. The old `index.ts` + `App.tsx` from US-001 were removed. Root layout lives at `app/_layout.tsx`; initial route at `app/index.tsx`.
- **Expo Go + SDK 54 + new-arch + `<Stack>` from expo-router throws at runtime**: `Exception in HostFunction: TypeError: expected dynamic type 'boolean', but had type 'string'`. For US-002 the root uses `<Slot />` as the minimal root. US-031 will revisit with a dev-client build (`expo run:ios`) where Stack + Tabs should work.
- **iOS simulator boot flow for mobile-mcp verification:** `xcrun simctl boot <name>` → `open -a Simulator` → `bunx expo start --ios` in background → `xcrun simctl openurl <udid> exp://127.0.0.1:8081` if Expo Go doesn't auto-open the project. mobile-mcp's `mobile_open_url` blocks `exp://` schemes by default, so shell out.
- **Supabase project:** `marketify` (ref=`zioiiruwdliftpmcumav`, region=`ap-southeast-1`). All MCP DB calls use this `project_id`. Pre-installed extensions: pgcrypto, uuid-ossp, pg_stat_statements, supabase_vault, pg_graphql. pg_cron + citext available on request.
- **Install extensions `with schema extensions`** (e.g. `create extension if not exists citext with schema extensions`) to keep advisors clean. `pg_cron` is the exception: Supabase requires it in `pg_catalog`, leave it unqualified.
- **Enum canonical list is docs/tech-architecture.md §4.2** (lines 399-427). After generating types, verify `src/types/supabase.ts` Constants block matches it exactly.
- **Local migration files live under `supabase/migrations/` with `YYYYMMDDHHMMSS_us_NNN_description.sql`.** Keep filename and MCP migration `name` (no timestamp prefix) consistent. Commit every applied migration — `list_migrations` is the DB side, the file is the source of truth for fresh applies.
- **`app_private` schema was created in US-003** via a companion migration (`us_003_app_private_schema`). Revoke is already in place for `anon` and `authenticated`. Triggers / security-definer functions in later stories should target this schema.
- **Reference `citext` as `extensions.citext` in every migration.** US-003 installed it with `with schema extensions`; the default `search_path` used by migrations does not include that schema, so an unqualified `citext` fails to resolve.
- **Supabase auto-enables RLS on new `public.*` tables** on this project. New table migrations will show up in `get_advisors({type:'security'})` as INFO-level `rls_enabled_no_policy` (not `rls_disabled`). These INFO findings are benign until policies land in US-009 — no commit-blocking action needed.
- **`listing_conditions` is a single data-driven table**, not separate pre/post tables. Discriminated by `condition_kind` enum ('pre' | 'post') per docs/tech-architecture.md §4.7 block 8. Pre-condition rows use metrics like `min_followers`; post-condition rows use `post_*` metrics. The story US-005 text named separate tables but the spec wins.
- **Circular FK pattern:** `listings.current_version_id` references `listing_versions(id)` but both tables are in the same migration. Create the `listings` table with the column nullable, create `listing_versions`, then `alter table public.listings add constraint ... foreign key ...`. Spec follows this pattern (§4.7 lines 659-661).
- **Supabase defaults views to `security_invoker=off` (effectively DEFINER)**; it will fail the `security_definer_view` advisor. Every view we create must be followed with `alter view <name> set (security_invoker = on)` to match the project standard at §4.1 line 375. Applied to `submission_reuse_view` in US-006.
- **US-009 TODO — `submission_reuse_view` correctness under RLS:** with `security_invoker=on`, the self-join on `public.submissions other` will be filtered by the caller's `submissions_lister_read` policy, so a lister's `reuse_count` will only count reuses on submissions *they themselves can see* — not cross-lister reuses, which is the ReuseBadge's entire purpose (spec §4.6). The spec comment at tech-architecture.md:760 said "Security-def view so RLS on base tables doesn't mask the count" but the body has no lister filter baked in — a spec contradiction. Resolution options: (a) replace the view with an `app_private` SECURITY DEFINER set-returning function that takes a submission_id and returns the count with base-table RLS bypassed; (b) keep the view but scope its input with a parameter. Must be addressed alongside the RLS migration in US-009 before the ReuseBadge UI (US-058, US-065) ships.
- **Trigger function hardening pattern (mandatory for every new trigger / PL/pgSQL function in this repo):**
  1. Declare `set search_path = ''` on the function (CREATE OR REPLACE drops prior SET clauses, so restate it on every redefinition).
  2. With empty `search_path`, qualify EVERY user-defined ref: tables `public.<tbl>`, types/enum casts `public.<type>` (e.g. `'tiktok'::public.platform`), `app_private.<fn>` for sibling functions.
  3. For pg_catalog **functions** (`now()`, `hashtextextended()`, `pg_advisory_xact_lock()`, etc.): bare is fine — `pg_catalog` is always implicitly searched even with empty `search_path`. Schema-qualifying as `pg_catalog.<fn>()` also works for true functions.
  4. **GREATEST / LEAST / COALESCE / NULLIF are SQL parser constructs, NOT functions.** They CANNOT be schema-qualified (`pg_catalog.greatest()` raises 42883). Always write them bare.
  5. SECURITY DEFINER functions especially require empty `search_path` — without it, a caller-controlled search_path can shadow `public.<table>` and divert writes. The advisor `function_search_path_mutable` flags this at WARN.
- **US-008 TODO — out-of-order Apify webhook delivery:** `app_private.denorm_metrics` uses `greatest(metrics_fetched_at, new.fetched_at)` to guard the timestamp column, but the surrounding column UPDATEs (e.g. `tiktok_follower_count`) are unconditional. If a webhook with an older `fetched_at` lands after a newer one (different `apify_run_id`, so the dedup index doesn't catch it), the trigger will overwrite fresh denorm columns with stale values even though the timestamp itself refuses to roll back. The spec §15 has the same gap. Resolution when US-008 (apify-core / webhook receiver) lands: either add `where new.fetched_at >= public.creator_profiles.metrics_fetched_at` to each branch's UPDATE, or guard the whole denorm block on a max(fetched_at) check inside the advisory lock. Update the spec section in the same PR.
- **US-008 nice-to-have — advisory lock collision risk:** `app_private.denorm_metrics` uses single-arg `pg_advisory_xact_lock(hashtextextended(...))`. Two distinct (social_link_id, scrape_mode) strings can collide to the same 64-bit key, causing unrelated creators to serialize unnecessarily. Not a correctness issue (only throughput). Standard fix is the two-arg form `pg_advisory_xact_lock(hashtext(social_link_id::text), hashtext(scrape_mode::text))`. Spec §15 codifies the single-arg form; address as a spec amendment when load matters.
