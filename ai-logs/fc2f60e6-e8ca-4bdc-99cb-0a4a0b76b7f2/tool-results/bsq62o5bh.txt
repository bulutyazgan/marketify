- **JWT helper contract (`supabase/functions/_shared/jwt.ts`):** `signJwt({sub, app_role, session_id}, {ttlSeconds?})` + `verifyJwt(token): JwtClaims`. HS256 via Web Crypto. Secret from `MARKETIFY_JWT_SECRET` (NOT the Supabase project JWT secret). - **Apify REST endpoint + slug encoding:** use `POST https://api.apify.com/v2/acts/<rest-slug>/runs?token=<APIFY_KEY>&waitForFinish=<secs>`. - **Deno tests for `_shared/` modules live alongside them as `<name>.test.ts` and run with `deno test --allow-env supabase/functions/_shared/<name>.test.ts`.** Assertions come from `jsr:@std/assert@1`. - **Public async wrappers must be declared `async` if they do any synchronous validation that might throw**, so the throw surfaces as a Promise rejection rather than a synchronous exception. `assertRejects` in `@std/assert` fails with "Function throws when expected to reject" if the function throws synchronously — a real footgun that can only be caught by actually running the test.
- **Constant-time dual-secret compare for rotation-aware webhook auth:** edge functions that authenticate third-party webhooks via a shared secret must (a) compare in constant time (XOR-accumulate over `max(len(a), len(b))` bytes, iterate the longer one), (b) check BOTH `<SECRET>` and `<SECRET>_PREVIOUS` env vars without short-circuit, and (c) return false if neither env var is set. The second point is subtle: `return timingSafeEqual(a, current) || timingSafeEqual(a, previous)` short-circuits when the first matches, which means an attacker who has recovered `current` can time-distinguish whether `previous` is configured. Correct shape is `let matchA = current ? timingSafeEqual(provided, current) : false; let matchB = previous ? timingSafeEqual(provided, previous) : false; return matchA || matchB` — both evaluated unconditionally. Reference: `supabase/functions/apify-webhook/index.ts` `verifyWebhookSecret`.

- **Reanimated reduced-motion pattern (callers + helpers):** `withReducedMotion(config, reduced)` in `src/design/motion.ts` spreads `reduceMotion: ReduceMotion.Always` onto any Reanimated `SpringConfig`/`TimingConfig` when the caller's `useReducedMotion()` is true. Reanimated v4 collapses `Always`-flagged animations to an instant snap on `onStart` — this is the sanctioned no-op; do NOT roll your own `withTiming(target, {duration: 0})` wrapper. Call shape: `withSpring(target, withReducedMotion<WithSpringConfig>(SPRING_SOFT, reduced))` — the **explicit `<WithSpringConfig>` generic is mandatory** because the presets are declared `as const satisfies WithSpringConfig` which narrows them to readonly literal types; under `strict` TS those literals don't satisfy the helper's `C extends {reduceMotion?: ReduceMotion}` constraint without widening. - **Animatable iOS shadow props in `useAnimatedStyle`: `shadowOpacity` + `shadowRadius` only — NOT `shadowOffset`.** `shadowOffset` is a compound `{width, height}` object prop that Reanimated 4 does not interpolate through `useAnimatedStyle`; the underlying UIKit shadow stays at its `StyleSheet` value and the offset never moves. - **Neubrutalist Pressable pattern (`src/components/primitives/Button.tsx`):** one shared value `pressed: 0 → 1` drives `translateX/Y`, `scale`, and `shadowOpacity` in a single `useAnimatedStyle`. `onPressIn` fires haptic + springs to 1, `onPressOut` springs to 0. - **Neubrutalist "lifted" tier is universal** — cards, buttons, chips, inputs, AND status pills all get `shadows.hard` from `src/design/tokens.ts` (offset 3,3, solid ink, opacity 1, radius 0). AuthProvider hydrates the cache once on mount via `hydrateTokenFromStorage()` and gates children on a `hydrated` boolean — returns `null` until the initial read settles to avoid a flash of logged-out state.
- **Supabase typed client for custom-JWT auth (`src/lib/supabase.ts`):** `createClient<Database>(url, anonKey, { auth: { persistSession: false, autoRefreshToken: false, detectSessionInUrl: false }, global: { fetch: marketifyFetch } })`. The three `auth.*: false` flags are required — otherwise Supabase-js tries to manage its own session storage and fights the custom `Authorization` header. The fetch wrapper builds headers via `new Headers(init?.headers)` which preserves Supabase-js's internally-set `apikey` header (required by the Supabase API gateway even for authed calls), then overwrites only `Authorization` when a token is cached. Unauthed reads work because the wrapper short-circuits to plain `fetch(input, init)` when no token is cached and Supabase-js's default `Authorization: Bearer <anon>` survives.
- **`app.config.ts` extends `app.json` dynamically** — `export default ({ config }: ConfigContext): ExpoConfig => ({ ...(config as ExpoConfig), extra: { ...config.extra, SUPABASE_URL: process.env.SUPABASE_URL, ... } })`. Expo CLI auto-loads `.env` into `process.env` when the dev server starts (no `dotenv` import needed). Values surface at runtime via `Constants.expoConfig?.extra`. Dynamic `app.config.ts` + static `app.json` coexist — Expo reads both and the dynamic file gets the merged config as its `config` argument. Required for the Supabase client wiring in US-030; future env surfaces (Supabase service-role keys, feature flags, etc.) add another entry under `extra`.
- **Custom `TabBar` replaces expo-router `<Tabs>` under Expo Go + SDK 54 new-arch.** The native `<Tabs>` navigator crashes at runtime with `"expected dynamic type 'boolean'"` (same root cause as US-002's `<Stack>` issue). Pattern: tab-group `_layout.tsx` renders `<View><Slot /></View><TabBar tabs={TABS} />`, and `TabBar` drives switching via `router.navigate(tab.path)` + `usePathname()`. `TabDef` MUST split `path` (filesystem form `/(group)/segment` for navigation) from `activePathname` (URL-stripped `/segment` for `usePathname` comparison) — route-group parens are invisible in the URL, so `pathname === path` never matches. Tabs use `accessibilityRole="tab"` with `accessibilityState={{ selected: focused }}`. `isActive` returns true on exact match OR nested match (`pathname.startsWith(tabPath + '/')`) so `/campaigns/new/step-1` still lights the Campaigns tab. Revisit with a dev-client build. Ref: `src/components/shared/TabBar.tsx` + `app/(creator)/_layout.tsx` + `app/(lister)/_layout.tsx`.
- **`__DEV__` tree-shaking only reaches code NESTED inside an `__DEV__`-branched component.** Metro dead-code removal keeps module-scope `const`s and imports even when no prod code references them — so dev-only data (fake tokens, seed UUIDs, test fixtures) MUST live inside a component function that only renders under `if (__DEV__) ...`. Pattern from `app/(auth)/index.tsx`: main component has `{__DEV__ ? <DevRolePicker /> : null}`, and `DevRolePicker` declares its fake-user constants inside its own function body. The closures (and everything they reference) then get stripped from the prod bundle together.
- **Typed auth context pattern (`src/lib/auth.tsx`):** `AuthUser = Database['public']['Tables']['users']['Row']` and `AuthRole = Database['public']['Enums']['user_role']` derive directly from generated Supabase types — no hand-maintained shapes. `useAuth()` returns `{ user, role: user?.role ?? null, signIn(token, user), signOut() }`. Validate shape when re-hydrating from storage (a structural `isAuthUserShape` guard, not just `JSON.parse as AuthUser`) so a stale/corrupted entry from a previous schema clears itself instead of propagating `undefined` fields into React state. `signIn`/`signOut` are fire-and-forget: they `setUser(...)` synchronously and `void persistToken(...)` in the background — the in-memory cache in storage.ts is updated synchronously inside `persistToken` so the next Supabase fetch sees the new JWT immediately. Optimistic-UI flow; if the app crashes between `setUser` and the SecureStore write settling, the user just signs in again on next launch.
- **`supabase.functions.invoke` error shape + decoding:** on a non-2xx, the v2 client returns `{ data: null, error: FunctionsHttpError }` where `error.context` is the raw `Response` object (NOT the parsed body). Pattern: type-narrow on the context (`typeof (ctx as {json?:unknown}).json === 'function'` + `'status' in ctx`), then `await (ctx as Response).json()` once to pull the `{error: CODE}` body. Do NOT `.clone()` first — supabase-js v2 does not pre-consume the error Response, so the stream is unread and `.clone()` just adds a second failure surface without benefit. Fallback path (non-JSON body) is a catch block that falls through to a generic toast. This is the canonical shape for every edge-function caller that needs to render inline conflict errors (US-033 uses it; US-034 creator signup will reuse verbatim). Ref: `app/(auth)/signup/lister.tsx` `onSubmit`.
- **Always-mount inline error Text for reliable Android `accessibilityLiveRegion` announcements.** Android's AccessibilityNodeInfo live-region semantics fire on CONTENT CHANGES of an existing node — a node that mounts at the same time as its first content does not reliably announce. Pattern: render the error `<Text>` unconditionally with `accessibilityLiveRegion="polite"` and toggle between the error string and `''` via the `error ?? ''` prop. Collapse the node visually with a `{ height: 0, opacity: 0, overflow: 'hidden' }` style when empty, and gate `importantForAccessibility` (`'yes'` when error present, `'no-hide-descendants'` when empty) so the focus order skips it while empty. iOS uses `accessibilityRole="alert"` on the parent or the error Text itself for the same behavior — `accessibilityLiveRegion` is Android-only (see existing pattern above). Applies to every form-field inline error, not just signup. Ref: `app/(auth)/signup/lister.tsx` `Field` subcomponent.
- **Forcing an unauthed app state for mobile-mcp verification requires BOTH app-terminate AND keychain reset before relaunch.** `expo-secure-store` is iOS Keychain backed; `xcrun simctl keychain <udid> reset` only clears the keychain persistence layer, not the in-memory AuthProvider state of a running app. If the app is merely backgrounded, the `cachedToken` module-scope variable in `src/lib/storage.ts` + the React `user` state in `AuthProvider` both survive — and on foreground the AuthProvider does NOT re-read SecureStore (hydration runs once on mount). Correct sequence: `mobile_terminate_app(host.exp.Exponent)` → `xcrun simctl keychain <udid> reset` → `mobile_launch_app(host.exp.Exponent)` → tap the project in Expo Go's Recently Opened. The re-launch then mounts a fresh AuthProvider which hydrates from the (now empty) keychain and renders the role picker. Required for any story that verifies an auth-gated screen by logging out from outside the app (US-033 duplicate-username 409 test, and whenever US-037/US-038 UI lands before a sign-out-in-app flow exists).
