- **RLS helper pattern (use for every policy that scopes to the caller):** two SQL `stable` helpers `public.current_user_id()` (returns `nullif(auth.jwt() ->> 'sub', '')::uuid`) and `public.current_user_role()` (returns `nullif(auth.jwt() ->> 'role', '')::public.user_role`). - **The `submission_reuse_view` → `submission_reuse_count(uuid)` function swap (US-009 resolution):** the view with `security_invoker=on` would be filtered by the caller's `submissions_lister_read` policy and hide cross-lister reuses (the ReuseBadge's entire purpose). Revoked from public, granted to authenticated + service_role. Keep this pattern for any other cross-tenant aggregate the UI needs.
- **Verify RLS via a transactional `begin; ... rollback;` block:** seed fresh fixtures with known `auth.uid`s, then `set local role authenticated` + `set local "request.jwt.claims" = '{"sub":"<uuid>","role":"creator"}'` and assert row visibility via `insert into _asserts ...` rows that you `select *` at the end before rolling back. Important: `grant insert, select on _asserts to authenticated` before switching roles, else you'll get `permission denied for table _asserts`. `PERFORM` is PL/pgSQL only — don't use it outside a DO block. Check `information_schema.columns` before writing fixtures so INSERTs match real schema (columns migrate between blocks — e.g. `listing_version_id` lives on `applications`, not `submissions`).
- **Trigger function hardening pattern (mandatory for every new trigger / PL/pgSQL function in this repo):**
  1. Declare `set search_path = ''` on the function (CREATE OR REPLACE drops prior SET clauses, so restate it on every redefinition).
  2. With empty `search_path`, qualify EVERY user-defined ref: tables `public.<tbl>`, types/enum casts `public.<type>` (e.g. `'tiktok'::public.platform`), `app_private.<fn>` for sibling functions.
  3. For pg_catalog **functions** (`now()`, `hashtextextended()`, `pg_advisory_xact_lock()`, etc.): bare is fine — `pg_catalog` is always implicitly searched even with empty `search_path`. Schema-qualifying as `pg_catalog.<fn>()` also works for true functions.
  4. **GREATEST / LEAST / COALESCE / NULLIF are SQL parser constructs, NOT functions.** They CANNOT be schema-qualified (`pg_catalog.greatest()` raises 42883). Always write them bare.
  5. SECURITY DEFINER functions especially require empty `search_path` — without it, a caller-controlled search_path can shadow `public.<table>` and divert writes. The advisor `function_search_path_mutable` flags this at WARN.

- **Apify REST endpoint + slug encoding:** use `POST https://api.apify.com/v2/acts/<rest-slug>/runs?token=<APIFY_KEY>&waitForFinish=<secs>`. REST slugs replace `/` with `~` (`clockworks~tiktok-scraper`, `apify~instagram-scraper`). `waitForFinish` makes Apify block up to N seconds before returning the run object — if the run exceeds N, Apify returns `status=RUNNING` and webhooks pick up the terminal state later. Response shape is `{ data: { id, defaultDatasetId, status, ... } }`. For this project the wrapper at `supabase/functions/_shared/apify.ts` uses direct REST (not the `apify-client` SDK) because Deno in Edge Runtime handles plain fetch cleanly and `Deno.env.get("APIFY_KEY")` is the sanctioned way to read the token.

Throw a fixed-string error with status + slug only; if you need the body for debugging, log it separately with redaction.

- **Public async wrappers must be declared `async` if they do any synchronous validation that might throw**, so the throw surfaces as a Promise rejection rather than a synchronous exception. `assertRejects` in `@std/assert` fails with "Function throws when expected to reject" if the function throws synchronously — a real footgun that can only be caught by actually running the test.
- **Constant-time dual-secret compare for rotation-aware webhook auth:** edge functions that authenticate third-party webhooks via a shared secret must (a) compare in constant time (XOR-accumulate over `max(len(a), len(b))` bytes, iterate the longer one), (b) check BOTH `<SECRET>` and `<SECRET>_PREVIOUS` env vars without short-circuit, and (c) return false if neither env var is set. The second point is subtle: `return timingSafeEqual(a, current) || timingSafeEqual(a, previous)` short-circuits when the first matches, which means an attacker who has recovered `current` can time-distinguish whether `previous` is configured. Correct shape is `let matchA = current ? timingSafeEqual(provided, current) : false; let matchB = previous ? timingSafeEqual(provided, previous) : false; return matchA || matchB` — both evaluated unconditionally. Reference: `supabase/functions/apify-webhook/index.ts` `verifyWebhookSecret`.

- **Apify transitional run statuses are READY, RUNNING, TIMING-OUT, ABORTING.** When `waitForFinish` expires before a run reaches a terminal state, Apify returns one of these four — in all four cases the run continues and the registered webhook will still fire the terminal event later. Map all four to 'refreshing' in any caller that derives a metrics-status from the immediate actor-run response; only `SUCCEEDED` should map to 'fresh' and any other (FAILED/TIMED-OUT/ABORTED) to 'failed'. Don't special-case only RUNNING.
- **Reanimated reduced-motion pattern (callers + helpers):** `withReducedMotion(config, reduced)` in `src/design/motion.ts` spreads `reduceMotion: ReduceMotion.Always` onto any Reanimated `SpringConfig`/`TimingConfig` when the caller's `useReducedMotion()` is true. Reanimated v4 collapses `Always`-flagged animations to an instant snap on `onStart` — this is the sanctioned no-op; do NOT roll your own `withTiming(target, {duration: 0})` wrapper. Call shape: `withSpring(target, withReducedMotion<WithSpringConfig>(SPRING_SOFT, reduced))` — the **explicit `<WithSpringConfig>` generic is mandatory** because the presets are declared `as const satisfies WithSpringConfig` which narrows them to readonly literal types; under `strict` TS those literals don't satisfy the helper's `C extends {reduceMotion?: ReduceMotion}` constraint without widening. Inline the helper inside `withSpring` rather than caching to a `const spring` — `reduced` can flip at runtime when the user toggles OS reduce-motion. `useReducedMotion()` in `src/design/useReducedMotion.ts` uses a module-scope `cached: boolean | null` so only the first hook mount per app session pays the async `AccessibilityInfo.isReduceMotionEnabled()` race — subsequent mounts read the cached value synchronously and never flash `false` on devices with reduce-motion on.
- **Animatable iOS shadow props in `useAnimatedStyle`: `shadowOpacity` + `shadowRadius` only — NOT `shadowOffset`.** `shadowOffset` is a compound `{width, height}` object prop that Reanimated 4 does not interpolate through `useAnimatedStyle`; the underlying UIKit shadow stays at its `StyleSheet` value and the offset never moves. For the neubrutalist "shadow collapses to (0,0) on press" spec (docs/design.md §1.5 line 118), the sanctioned pattern is to **animate `shadowOpacity` from 1 → 0** while simultaneously translating the view `(0,0) → (3,3)` — the shadow visually disappears into the pressed-down button. Reference: `src/components/primitives/Button.tsx`.
- **Neubrutalist Pressable pattern (`src/components/primitives/Button.tsx`):** one shared value `pressed: 0 → 1` drives `translateX/Y`, `scale`, and `shadowOpacity` in a single `useAnimatedStyle`. `onPressIn` fires haptic + springs to 1, `onPressOut` springs to 0. Guard BOTH handlers with `if (!interactive) return;` (the disabled-state `styles.disabled` clears shadowOpacity, but `handlePressOut` still dispatches a spring if unguarded — stale-closure hygiene). `interactive = !disabled && !loading` — loading implies the press is blocked too. Pressable's own `disabled={!interactive}` prevents touch events from reaching RN when disabled, but the guards are defense-in-depth. Every primitive that needs a press animation (CampaignCard, Chip, BottomSheet drag handle) should reuse this shape.
- **Toggle-chip accessibility: `accessibilityRole="checkbox"` with `accessibilityState={{ checked }}`, NOT `"button"` with `{ selected }`.** `role="button"` ignores `aria-selected` — most screen readers silently drop it, so a filter chip reads as an inert button and the user can't tell what's on vs off. `checkbox` maps cleanly to `aria-checked` on iOS/Android and the `checked` boolean surfaces as "checked"/"not checked" in VoiceOver/TalkBack. Applies to any toggleable pill (filter chips, segmented controls, multi-select tags). Reference: `src/components/primitives/Chip.tsx`. For read-only `View`s (e.g. StatusPill), do NOT set `accessibilityRole="text"` — that role is for `Text` nodes only and iOS silently rejects it on a `View`; instead use `accessible` + `accessibilityLabel` on the `View` and let the child `Text` inherit silently.
- **`textStyles.micro` is the single source of truth for pill/chip labels** — Rubik_700Bold (stand-in for Satoshi 700 per the US-023 fallback chain), 11pt, lineHeight 14, `letterSpacing: 0.88`, `textTransform: 'uppercase'`. The 0.88 is the absolute-pt conversion of §1.3's `0.08em at 11pt` (RN letterSpacing is absolute points, not em). Every pill/chip/status-badge should apply `textStyles.micro` plus its semantic color inline, plus `allowFontScaling={false}` per the §8 checklist rule that UI chrome opts out of Dynamic Type. Added in US-026 (the first story that needed it); US-023 intentionally deferred micro since its AC enumerated only 6 keys.
