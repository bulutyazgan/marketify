## Codebase Patterns
*Append-only consolidated learnings. Each future iteration reads this BEFORE doing anything.*

- **Authoritative specs are `docs/product-plan.md`, `docs/design.md`, `docs/tech-architecture.md`.** When the story is ambiguous, read the relevant section there. Do not invent product behavior.
- **Use Supabase MCP, not the supabase CLI**, for migrations, queries, edge functions, and type generation. Migrations are append-only and named `us_NNN_description`.
- **Use mobile-mcp for any UI verification.** Never claim a UI story is done without a screenshot from `mobile_take_screenshot`.
- **Use the `feature-dev:code-reviewer` subagent before every commit.** It catches drift across iterations that you can't see in your fresh context.
- **Delegate exploration.** If a story would have you read >3 files, spawn the `Explore` subagent first and work from its summary.
- **Always regenerate `src/types/supabase.ts`** after any schema migration. Commit the regenerated file in the same commit as the migration.
- **Use `bun`, not npm/yarn/pnpm.** It is what's installed locally.
- **JWT secret is `MARKETIFY_JWT_SECRET`**, separate from Supabase's built-in JWT secret. We sign our own per docs/product-plan.md §2.
- **Apify key is `APIFY_KEY`** in `.env`. Read via `process.env.APIFY_KEY` or `Deno.env.get("APIFY_KEY")`. Never paste into source.
- **Edge functions live in `supabase/functions/<name>/index.ts`** with shared utils under `supabase/functions/_shared/`.

- **Circular FK pattern:** `listings.current_version_id` references `listing_versions(id)` but both tables are in the same migration. Create the `listings` table with the column nullable, create `listing_versions`, then `alter table public.listings add constraint ... foreign key ...`. Spec follows this pattern (§4.7 lines 659-661).
- **Supabase defaults views to `security_invoker=off` (effectively DEFINER)**; it will fail the `security_definer_view` advisor. Every view we create must be followed with `alter view <name> set (security_invoker = on)` to match the project standard at §4.1 line 375. Applied to `submission_reuse_view` in US-006.
- **RLS helper pattern (use for every policy that scopes to the caller):** two SQL `stable` helpers `public.current_user_id()` (returns `nullif(auth.jwt() ->> 'sub', '')::uuid`) and `public.current_user_role()` (returns `nullif(auth.jwt() ->> 'app_role', '')::public.user_role`). Both use `set search_path = ''` and schema-qualify every ref. `nullif` on the empty string makes a missing claim safely return NULL (all `id = current_user_id()` checks fail closed); for `current_user_role()` this is a hardening deviation from the spec's verbatim `(auth.jwt() ->> 'role')::user_role` — intentional, keeps the fn from raising on malformed JWTs. **Note the `app_role` claim (not `role`)** — the bare `role` claim is reserved by PostgREST for the Postgres DB role (always `"authenticated"` on our tokens); see Pattern #JWT-Claims below for why. Policies call these helpers instead of inlining `auth.jwt()` for readability + single-source-of-truth. Ref: `supabase/migrations/20260418020000_us_jwt_app_role_claim.sql` (supersedes the `role`-claim version from `us_009_rls_policies.sql`).
- **The `submission_reuse_view` → `submission_reuse_count(uuid)` function swap (US-009 resolution):** the view with `security_invoker=on` would be filtered by the caller's `submissions_lister_read` policy and hide cross-lister reuses (the ReuseBadge's entire purpose). The fix: **drop the view** and replace with a SECURITY DEFINER function `public.submission_reuse_count(p_submission_id uuid) returns integer` that (1) does its own ownership check via join through `submissions → applications → listings.lister_id = current_user_id()`, (2) returns NULL when the caller isn't the listing owner, (3) bypasses RLS to count cross-lister reuses. Revoked from public, granted to authenticated + service_role. Keep this pattern for any other cross-tenant aggregate the UI needs.
- **Verify RLS via a transactional `begin; ... rollback;` block:** seed fresh fixtures with known `auth.uid`s, then `set local role authenticated` + `set local "request.jwt.claims" = '{"sub":"<uuid>","role":"creator"}'` and assert row visibility via `insert into _asserts ...` rows that you `select *` at the end before rolling back. Important: `grant insert, select on _asserts to authenticated` before switching roles, else you'll get `permission denied for table _asserts`. `PERFORM` is PL/pgSQL only — don't use it outside a DO block. Check `information_schema.columns` before writing fixtures so INSERTs match real schema (columns migrate between blocks — e.g. `listing_version_id` lives on `applications`, not `submissions`).
- **Trigger function hardening pattern (mandatory for every new trigger / PL/pgSQL function in this repo):**
  1. Declare `set search_path = ''` on the function (CREATE OR REPLACE drops prior SET clauses, so restate it on every redefinition).
  2. With empty `search_path`, qualify EVERY user-defined ref: tables `public.<tbl>`, types/enum casts `public.<type>` (e.g. `'tiktok'::public.platform`), `app_private.<fn>` for sibling functions.
  3. For pg_catalog **functions** (`now()`, `hashtextextended()`, `pg_advisory_xact_lock()`, etc.): bare is fine — `pg_catalog` is always implicitly searched even with empty `search_path`. Schema-qualifying as `pg_catalog.<fn>()` also works for true functions.
  4. **GREATEST / LEAST / COALESCE / NULLIF are SQL parser constructs, NOT functions.** They CANNOT be schema-qualified (`pg_catalog.greatest()` raises 42883). Always write them bare.
  5. SECURITY DEFINER functions especially require empty `search_path` — without it, a caller-controlled search_path can shadow `public.<table>` and divert writes. The advisor `function_search_path_mutable` flags this at WARN.
- **US-008 TODO — out-of-order Apify webhook delivery:** `app_private.denorm_metrics` uses `greatest(metrics_fetched_at, new.fetched_at)` to guard the timestamp column, but the surrounding column UPDATEs (e.g. `tiktok_follower_count`) are unconditional. If a webhook with an older `fetched_at` lands after a newer one (different `apify_run_id`, so the dedup index doesn't catch it), the trigger will overwrite fresh denorm columns with stale values even though the timestamp itself refuses to roll back. The spec §15 has the same gap. Resolution when US-008 (apify-core / webhook receiver) lands: either add `where new.fetched_at >= public.creator_profiles.metrics_fetched_at` to each branch's UPDATE, or guard the whole denorm block on a max(fetched_at) check inside the advisory lock. Update the spec section in the same PR.
- **US-008 nice-to-have — advisory lock collision risk:** `app_private.denorm_metrics` uses single-arg `pg_advisory_xact_lock(hashtextextended(...))`. Two distinct (social_link_id, scrape_mode) strings can collide to the same 64-bit key, causing unrelated creators to serialize unnecessarily. Not a correctness issue (only throughput). Standard fix is the two-arg form `pg_advisory_xact_lock(hashtext(social_link_id::text), hashtext(scrape_mode::text))`. Spec §15 codifies the single-arg form; address as a spec amendment when load matters.
- **`public.events` append-only is a social contract, not a DB-level guarantee.** US-008 reproduces the spec literally (`revoke all on public.events from anon, authenticated`), but `service_role` and `postgres` retain every grant including UPDATE/DELETE — by design per spec §4.4 lines 1083–1085 ("Service role only. No application read/write policy"). Any edge function or trigger that writes to `events` must treat INSERT as the only safe verb; there is no DB trigger that blocks UPDATE/DELETE. If stronger enforcement is ever desired, add a BEFORE UPDATE OR DELETE trigger raising `RAISE EXCEPTION` (this would also block maintenance scripts — considered too restrictive for v1).
- **In PL/pgSQL with `set search_path = ''`, `text[] || 'literal'` resolves to `anyarray || anyarray` and crashes with `22P02 malformed array literal`.** Postgres chooses the array+array overload for unknown-type literals. Always cast the right-hand side: `changed_fields || 'price_cents'::text`. Typed text columns (e.g. `new.version_bump_reason`) work bare because the operator resolves to `anyarray || anyelement` when the right side has a concrete element type. This gotcha is independent of the `public.<type>` schema-qualification pattern already in Codebase Patterns. Resolved in US-010 via `us_010_fix_array_append` follow-up migration.

Those columns are populated by §15b (`app_private.refresh_listing_thresholds` on `listing_conditions` + `app_private.refresh_thresholds_on_version_bump` on `listings.current_version_id`) which was intentionally scoped OUT of US-010 per its AC. - **`pg_net.net.http_post(url := NULL, ...)` is a silent no-op, not an error.** When the URL or any required string is NULL (e.g. a `vault.decrypted_secrets` lookup misses), pg_net returns a request id without raising and `cron.job_run_details` logs the run as `succeeded`. Always wrap pg_net calls inside cron bodies in a PL/pgSQL `do $body$ ... $body$` block that resolves required arguments first and `RAISE EXCEPTION` when any are NULL — that surfaces the misconfiguration in `cron.job_run_details` instead of pretending the job ran. Pattern documented in `supabase/migrations/20260417163100_us_011_fix_silent_no_secret.sql`.

Migrations DO NOT create the secrets — the actual key material is set out-of-band via the Supabase dashboard. Migrations document the required secret names in the header comment so the next operator knows what to populate. Pre-population, the cron raises and the failure is visible in `cron.job_run_details`. - **`pg_cron` job-name idempotency requires a guarded `cron.unschedule`.** `cron.schedule` raises on duplicate jobname; `cron.unschedule(text)` raises when the name is unknown. - **Edge functions are Deno, not RN — exclude `supabase/` from the Expo toolchain.** tsc (`tsconfig.json` exclude) and eslint (`eslint.config.js` ignores) both skip `supabase/**`. Edge-function code imports `Deno.env`, uses Web Crypto, and runs under Supabase Edge Runtime. Caveat: tsc `exclude` does not prevent files pulled in via imports, so app code must never import from `supabase/functions/*`. Any cross-boundary types should be re-declared or lifted to `src/types/`.
- **JWT helper contract (`supabase/functions/_shared/jwt.ts`):** `signJwt({sub, app_role, session_id}, {ttlSeconds?})` + `verifyJwt(token): JwtClaims`. HS256 via Web Crypto. Secret from `MARKETIFY_JWT_SECRET` (NOT the Supabase project JWT secret). Claims are `{sub, role: "authenticated", app_role: "creator"|"lister", session_id, iat, exp}`. `session_id` is spec-required per §4.4 (story AC said only 4 claims; spec wins). **`role` is ALWAYS `"authenticated"`** — it names the Postgres DB role PostgREST switches to. The app-level creator/lister distinction is on a separate `app_role` claim. `verifyJwt` enforces `alg=HS256`, `role === "authenticated"`, `app_role ∈ {creator, lister}`, exp > now, iat ≤ now+60s, and validates claim shapes. Any JSON/base64/atob malformation surfaces as `"Malformed JWT"`. Uses `crypto.subtle.verify` (not re-sign + compare) with a verify-only-usage key. Edge-function gate checks read `claims.app_role !== "creator"` (or `"lister"`), never `claims.role`. RLS helpers read `auth.jwt() ->> 'app_role'` (see RLS helper pattern above).
- **CORS headers shared module (`supabase/functions/_shared/cors.ts`):** `corsHeaders` allows `*` origin — safe because edge functions authenticate via `Authorization: Bearer <marketify_jwt>`, not cookies. Every edge function must splat this into responses and handle an `OPTIONS` pre-flight with `{ status: 204, headers: corsHeaders }`.
- **Atomic multi-insert edge-function flow uses a PL/pgSQL RPC, not back-to-back supabase-js inserts.** Two `supabase.rpc`/`.insert()` calls are two separate PostgREST transactions, so a mid-flight crash leaves orphan rows. The established pattern (US-014) is a single `public.<flow>_rpc` PL/pgSQL function invoked via `supabase.rpc(...)` from the edge function — PL/pgSQL wraps the body in an implicit transaction. Catch `unique_violation` with `GET STACKED DIAGNOSTICS v_constraint = constraint_name` and return `{error: CODE}` jsonb; let everything else `RAISE` so the edge function can log+500. Implicit-constraint names (`<table>_<column>_key`) are Postgres defaults — ALWAYS comment the expected names in the exception block, since a future migration that renames them would silently downgrade 409s to 500s. Revoke execute from anon/authenticated/public; grant only to service_role (no SECURITY DEFINER needed — service_role already has BYPASSRLS).
- **Supabase edge-function secrets are set via `supabase secrets set --env-file <tmpfile> --project-ref <ref>`, not via Supabase MCP.** The MCP has no secrets tool. Create a temp file containing `NAME=value`, run the CLI, delete the temp file. Never echo the secret value into chat. After setting, re-invoke the edge function — secret names are available via `Deno.env.get(...)`. Running the edge function before the secret is set surfaces as a 500 at the point the helper tries to read it (for `MARKETIFY_JWT_SECRET`, that's `signJwt` → `"MARKETIFY_JWT_SECRET is not set"` → caught → returned as `{error: "JWT_ERROR"}` in the current auth-* pattern).
- **Edge-function smoke tests live under `scripts/test/<fn-name>.ts`, invoked with `bun run`.** Read `SUPABASE_URL` + `SUPABASE_ANON_KEY` from `process.env` (bun auto-loads `.env`). Send `Authorization: Bearer <anon>` + `apikey: <anon>` headers even for `verify_jwt=false` functions — the Supabase gateway still requires a valid apikey at the edge. Use `Date.now()` for unique test fixture values so repeat runs don't collide. Every conflict branch (e.g. 409 USERNAME_TAKEN vs 409 EMAIL_TAKEN) gets its own case — the constraint-name mapping in the RPC's exception handler is only validated by cases that actually hit each branch. Clean up test rows via `execute_sql` after each run.
- **Anonymous `unique` declarations in CREATE TABLE get Postgres default names (`<table>_<column>_key`)** — e.g. `username citext not null unique` on `public.users` produces `users_username_key`. Any PL/pgSQL that inspects `constraint_name` depends on this. Verify via `select conname from pg_constraint where conrelid = 'public.<table>'::regclass` before hardcoding a name. If you later want to rename, drop+re-add with explicit `constraint <name> unique (col)` — and update every dependent exception handler in the same migration.
- **Apify REST endpoint + slug encoding:** use `POST https://api.apify.com/v2/acts/<rest-slug>/runs?token=<APIFY_KEY>&waitForFinish=<secs>`. REST slugs replace `/` with `~` (`clockworks~tiktok-scraper`, `apify~instagram-scraper`). `waitForFinish` makes Apify block up to N seconds before returning the run object — if the run exceeds N, Apify returns `status=RUNNING` and webhooks pick up the terminal state later. Response shape is `{ data: { id, defaultDatasetId, status, ... } }`. For this project the wrapper at `supabase/functions/_shared/apify.ts` uses direct REST (not the `apify-client` SDK) because Deno in Edge Runtime handles plain fetch cleanly and `Deno.env.get("APIFY_KEY")` is the sanctioned way to read the token.
- **Never echo external API error bodies into thrown Error strings when the request URL carries a secret in the query string.** Apify (and several other providers) sometimes include the request URL in error payloads, so `throw new Error(\`... ${body}\`)` can land the token in edge-function logs. Throw a fixed-string error with status + slug only; if you need the body for debugging, log it separately with redaction.
- **Deno tests for `_shared/` modules live alongside them as `<name>.test.ts` and run with `deno test --allow-env supabase/functions/_shared/<name>.test.ts`.** Assertions come from `jsr:@std/assert@1`. Mock fetch by replacing `globalThis.fetch` with a spy that records `{url, init}` and returns a canned `Response`; always restore in a `try/finally`. Deno isn't on every dev machine; `brew install deno` on macOS is a one-liner. If Deno isn't available in a given iteration, document it in the progress entry and have the CI for this run the tests.
- **Public async wrappers must be declared `async` if they do any synchronous validation that might throw**, so the throw surfaces as a Promise rejection rather than a synchronous exception. `assertRejects` in `@std/assert` fails with "Function throws when expected to reject" if the function throws synchronously — a real footgun that can only be caught by actually running the test.
- **Constant-time dual-secret compare for rotation-aware webhook auth:** edge functions that authenticate third-party webhooks via a shared secret must (a) compare in constant time (XOR-accumulate over `max(len(a), len(b))` bytes, iterate the longer one), (b) check BOTH `<SECRET>` and `<SECRET>_PREVIOUS` env vars without short-circuit, and (c) return false if neither env var is set. The second point is subtle: `return timingSafeEqual(a, current) || timingSafeEqual(a, previous)` short-circuits when the first matches, which means an attacker who has recovered `current` can time-distinguish whether `previous` is configured. Correct shape is `let matchA = current ? timingSafeEqual(provided, current) : false; let matchB = previous ? timingSafeEqual(provided, previous) : false; return matchA || matchB` — both evaluated unconditionally. Reference: `supabase/functions/apify-webhook/index.ts` `verifyWebhookSecret`.
- **Third-party-webhook-receiver edge functions deploy with `verify_jwt=false`** (Supabase MCP `deploy_edge_function({verify_jwt: false})`). The provider cannot send an `Authorization: Bearer <marketify-jwt>` — authentication is via the provider's own header (e.g. `X-Apify-Webhook-Secret`). The Supabase API gateway still requires a valid project `apikey` though, so smoke tests must send `Authorization: Bearer <anon>` + `apikey: <anon>` headers in addition to the webhook-specific auth header. For Apify specifically, pair this with a `payloadTemplate` in the per-run webhook registration that injects `social_link_id` / `scrape_mode` / `run_id` / `apify_finished_at` — these fields are NOT in Apify's default event shape and are required by our receiver contract.
- **Webhook-receiver idempotency via service-role RPC + ON CONFLICT DO NOTHING:** the receiver's persistence step goes through a PL/pgSQL RPC that `INSERT ... ON CONFLICT (<run-id-col>) WHERE <run-id-col> IS NOT NULL DO NOTHING` and returns jsonb `{inserted: bool, duplicate: bool, <id>?: uuid}`. Paired with a partial unique index on the run-id column, replays of the same webhook are safe. Revoke the RPC from anon/authenticated/public and grant only to service_role. The RPC is invoked by the edge function with a service-role Supabase client — NO `SECURITY DEFINER` needed because service_role already has BYPASSRLS. Reference: `public.apify_webhook_persist_tiktok_profile` + `metric_snapshots_run_uniq` (US-006/016).
- **Scope env-var reads to the branch that actually needs them.** A single top-of-handler `Deno.env.get("OPTIONAL_KEY")` guard that 500s on missing keys will trip branches that don't need the key. For the apify-webhook, `APIFY_KEY` is only needed to fetch the dataset on the SUCCEEDED branch — gating the whole handler on it broke the FAILED/TIMED_OUT/ABORTED branches (which have no dataset). Move the lookup inside the conditional, and only return `SERVER_MISCONFIGURED` when you actually can't proceed.
- **`supabase/functions/<name>-live.ts` convention for Apify-touching e2e tests:** any smoke test that spends real Apify credits lives alongside the cheap-path smoke as `<fn>-live.ts`, is NOT part of the default test suite, and documents what it costs + what external state it touches. US-016 used `clockworks~tiktok-scraper` against `@tiktok` (cheap + stable + public) with seed `social_link_id = 11111111-1111-1111-1111-111111111020` — good reference fixture for future Apify-touching stories. DB-side verification (snapshot row + denorm trigger) is done separately via MCP `execute_sql` after the script completes.
- **Apify `payloadTemplate` requires `shouldInterpolateStrings: true` for quoted `"{{placeholder}}"` substitution.** Without this flag, quoted placeholders stay literal (`"{{resource.id}}"` delivered as-is) and the receiver's validation layer 400s. Bare unquoted `{{var}}` substitutes raw JSON but breaks `JSON.stringify(payloadTemplate)` because the placeholder isn't valid JSON. Rule: every webhook spec we build has `shouldInterpolateStrings: true` and uses quoted placeholders exclusively. Reference: `supabase/functions/_shared/apify.ts` `ApifyWebhookSpec` + `supabase/functions/auth-signup-creator/index.ts`.
- **Apify transitional run statuses are READY, RUNNING, TIMING-OUT, ABORTING.** When `waitForFinish` expires before a run reaches a terminal state, Apify returns one of these four — in all four cases the run continues and the registered webhook will still fire the terminal event later. Map all four to 'refreshing' in any caller that derives a metrics-status from the immediate actor-run response; only `SUCCEEDED` should map to 'fresh' and any other (FAILED/TIMED-OUT/ABORTED) to 'failed'. Don't special-case only RUNNING.

The `deploy_edge_function` `files` array must include every `_shared/*.ts` dependency, not just the new one. Ref: `_shared/eligibility.ts` shared by `get-listing-detail` (v4) + `apply-to-listing` (v2).
- **`KeyboardAvoidingView` is unreliable inside an RN `<Modal>` (and therefore inside our `BottomSheet`).** The Modal opens its own native window, so KAV's measured frame and the keyboard's reported frame disagree and the contents do not get pushed up. Replace KAV with a direct `Keyboard.addListener('keyboardWillShow' / 'keyboardWillHide')` (use `keyboardDidShow`/`keyboardDidHide` on Android) that drives `paddingBottom` on the inner `<View>` from `e.endCoordinates.height`. Remember that `BottomSheet` snap point must accommodate `keyboard_height (~336px iOS) + content_height` — for the apply sheet that meant raising `snapPoints` from `[0.7]` to `[0.85]`. Ref: `src/screens/listing-detail/ApplySheet.tsx`.
- **Defer `router.replace`/`router.push` after dismissing a `BottomSheet` by ~350ms** so the sheet's spring-out animation (`SPRING_SNAPPY` settles in ~300ms — see `src/design/motion.ts`) plays cleanly before the new screen mounts. Without the delay, the sheet slides down on top of the freshly-rendered destination, which looks broken. The same delay applies to any imperative navigation triggered from a Modal/BottomSheet success path. Ref: `app/(creator)/listing/[id].tsx` `onApplied`.
- **Run cheap checks before upstream oEmbed/scraper calls in edge functions; return `502 OEMBED_UNAVAILABLE` (not 500) when the upstream itself fails.** Order inside an endpoint that relies on a third-party URL validator: JWT → role → parse body → regex-classify URL → DB lookups → status/permission gates → affirmation/eligibility check → upstream oEmbed → RPC. This is both a cost optimization (don't burn TikTok rate-limit slots on requests that will 4xx regardless) and a DDoS-amplifier guard (an unauth POST must not be able to force an upstream fetch). Split the error codes: `500 SERVER_MISCONFIGURED` / `500 DB_ERROR` for our-fault failures, `502 OEMBED_UNAVAILABLE` for upstream unavailability — clients that auto-retry benefit because upstream outages are usually retryable, DB bugs rarely are. Document BOTH codes in the contract comment. Ref: `supabase/functions/submit-video/index.ts` oEmbed section + error envelope header.
- **Defensive null-guards on NOT NULL columns fetched via service-role client, when the row feeds a security-relevant gate.** The database schema can enforce NOT NULL, but a TypeScript read from a future mis-migrated row that returns `null` will silently propagate into downstream logic. When the value gates a security check (e.g. `application.listing_version_id` → the post-condition affirmation check; an empty `.eq('listing_version_id', null)` fetch returns zero rows → no affirmations required → gate bypassed), add an explicit `if (!row.field) { 500 DB_ERROR }`. One extra branch is cheaper than a future exploitable silent-pass. Ref: `supabase/functions/submit-video/index.ts` listing_version_id guard.
- **PostgREST embeds silently NULL joined columns when the target table's RLS filters the caller out — always audit every `.select('..., related!fk(col)')` embed against the related table's RLS policies.** Failure mode: the embed syntax succeeds, returns rows, and populates the parent columns; the embedded field just comes back as `null` or `[]` under RLS because PostgREST applies the policies per-row transparently. Highest-risk embeds on this project: (a) `users!lister_id(username)` where `users_self_select` is self-only — every cross-tenant handle read (creator viewing lister handle, lister viewing applicant handle) will null out; (b) `listing_versions!...(...)` where `listing_versions_read_if_listing_readable` gates on `listings.status='active' OR caller=lister` — any historical/closed listing version the caller doesn't own will null; (c) `listings!...(title)` where `listings_public_read` requires `status='active'` — same closed-listing nullout. Rule: before writing a PostgREST embed, check the target table's RLS; if the policy wouldn't return the specific row under the caller's role, replace the embed with a SECURITY DEFINER RPC that does its own ownership check. DO NOT widen the self-only policy to "return usernames if handle is public" — that leaks the full row shape; DEFINER + narrow column projection is the correct primitive. Ref: Codebase Pattern #40 (`submission_reuse_count`), Codebase Pattern #110 (`apply_to_listing_rpc`), and `public.list_my_applications()` from US-043.
- **SECURITY DEFINER "list-my-rows-with-foreign-context" RPC skeleton.** Contract: `returns table(<flat columns including the cross-tenant bits>)`, `language sql stable security definer set search_path = ''`, body is a straight `select ... from public.<owned_table> t left join public.<foreign_table> f on ... where t.<owner_col> = public.current_user_id() order by ...`. Grants: `revoke all ... from public`, `revoke all ... from anon`, `grant execute ... to authenticated`. Service-role is intentionally not granted — it already has BYPASSRLS and wouldn't use this shape. Why `stable`, not `volatile`: the function reads but doesn't write, so PostgREST can plan around it and query planner caches can reuse results within a statement. Why `sql` over `plpgsql`: the body is one statement, no control flow, no GET STACKED DIAGNOSTICS — inlines better. Verified pattern via a transactional `begin; set local role authenticated; set local "request.jwt.claims" = '{"sub":"<uuid>","role":"creator"}'; select * from public.<fn>();` block that confirms (a) the DEFINER bypass returns the cross-tenant columns, (b) the ownership check still gates other users' rows (swap the sub claim to a different creator and assert zero rows). Ref: `supabase/migrations/20260418010100_us_043_list_my_applications_rpc.sql`. Applies to any future "my X with lister/creator/reviewer context" list screen.
- **Screen-RPC naming convention: `public.list_my_<things>` for caller-scoped list reads, `public.<action>_rpc` for mutations, `public.<thing>_<metric>(arg)` for aggregates.** Precedents: `public.list_my_applications()` (US-043) / `public.apply_to_listing_rpc(...)` (US-041) / `public.submission_reuse_count(uuid)` (US-009). Keep the `list_my_*` functions argumentless (the caller is implicit via `current_user_id()`) — passing a caller UUID would re-introduce the spoofing surface the helpers exist to eliminate. Keep them in the `public` schema (not `app_private`) because they need to be callable via PostgREST `supabase.rpc(...)`; `app_private` is reserved for server-side triggers + cron bodies. The `authenticated` grant is mandatory — without it PostgREST returns 42501 "permission denied for function" and the MCP `execute_sql` test path (runs as `postgres`) silently passes, masking the bug.
- **Segmented control a11y: `accessibilityRole="tablist"` wrapper + `accessibilityRole="tab"` on each button + `accessibilityState={{ selected }}`**, NOT the Chip primitive's `role="checkbox"` + `checked`. Chip is for multi-select filter rows where each pill is independent; a segmented control is single-select and semantically is a tab bar. iOS VoiceOver reads "tab, Pending, selected, 1 of 4" for the tablist shape — correct under Apple's a11y guidance. `selected` is the right state (not `checked`) because the user isn't toggling a boolean, they're picking one of N mutually-exclusive views. Inline the control rather than promoting to a shared primitive until a second screen needs it — premature abstraction hurts more than a ~40-line duplicate. Ref: `SegmentControl` in `app/(creator)/applications.tsx` (US-043).
- **Enum-to-UI-bucket maps live as a static `Record<DBEnum, UISegment>` next to the screen that owns the mapping, not in a shared constants file.** `application_status` has 6 values; the Applied tab shows 4 segments (pending / approved / rejected / cancelled), with `withdrawn + cancelled_listing_edit + cancelled_listing_closed` all bucketing to "cancelled" per design.md §3.1 label + the AC's 4-segment design. Do NOT collapse the three cancellation variants in the DB (they carry distinct semantic meaning for future admin/analytics use); collapse only at the UI layer. - **Caller-scoped dashboard aggregate RPC beats N round-trips.** When a screen needs multiple `count(*)` aggregates all gated by the same caller-scoped filter (Lister Home: active campaigns / pending applications / pending submissions), expose ONE `SECURITY INVOKER` SQL function returning a typed row with all the counts rather than N PostgREST `.select('*', { count: 'exact', head: true })` calls. Three wins: (1) atomic — the numbers come from the same snapshot, no race where they disagree because they were fetched 20ms apart; (2) typed via `Database['public']['Functions']['<name>']['Returns'][number]` so the caller reads column names, not Range-header counts; (3) one network hop. - **`allowFontScaling={false}` vs `maxFontSizeMultiplier={1.3}` — they are NOT interchangeable.** Per docs/design.md §1.3 + §11 QA checklist: CONTENT text (body, captions, display-size numbers in stat tiles, card titles, anything informational) must honor system Dynamic Type up to 130% via `maxFontSizeMultiplier={1.3}`. UI CHROME (pills, chips, tab-bar labels, segment buttons — fixed-sized containers where scaling would break layout) opts out entirely with `allowFontScaling={false}`. The trap: a hero metric using `textStyles.display` LOOKS like chrome (big, standalone, inside a tile) but is CONTENT — the whole point of the screen. Getting this wrong makes the app inaccessible to users with large system text without any compile-time signal. - **Zero-data state and empty-state are distinct UI states even when they share the same data shape.** A dashboard that renders `0 / 0 / 0` across three tiles when the user has nothing to show is a UX miss — design.md §5 specifies dedicated empty-state copy for every screen (Lister dashboard: "Post your first bounty."). Don't conflate "API returned successfully" with "there is data worth showing".
