When flipping between distinct UI screens inside a single route (form ‚Üí loading ‚Üí navigate), the incoming screen's primary status Text/View should carry `accessibilityRole='header' + accessibilityLiveRegion='polite'` to announce the transition. iOS VoiceOver re-reads the screen on mount so this also helps but is not strictly required there.
  - **Avoid the `didSucceed` finally-guard on navigation-triggering submits.** The temptation is real: `router.replace` unmounts the screen, so `setSubmitting(false)` in `finally` "could flash the form for a frame". But `router.replace` is synchronous enough under React Native that no flash materialized in testing, and React suppresses state updates on unmounted components. Meanwhile the guard makes the form unrecoverable if the navigation itself throws (rare but real ‚Äî deep-link intercepts, a crash in the destination layout, a back-gesture race). Trust the reconciler; always call `setSubmitting(false)` in `finally`.
  - The existing `mobile_type_keys` first-char-drop gotcha (#101) also drops chars mid-string, not just the first char ‚Äî observed `u034_both_1719`‚Üí`u34_both_1719`, `tiktok`‚Üí`tktok`, `instagram`‚Üí`istagram` in a single two-handle fill. Tests that don't care about handle *accuracy* (e.g. this loading-state screenshot just needs both fields non-empty) can tolerate the drops; tests that DO care (e.g. duplicate-username 409, real Apify scrape) must verify via `mobile_list_elements_on_screen` after each type and retype via the Select All menu.
  - SkeletonCard at `height={96}` reads as a "compact stat row" placeholder in the loading swap; the default 140px was too tall relative to two stacked cards + a heading. `height` is a pure visual knob ‚Äî consider exposing other compact heights (72? 88?) when future stories (US-035 creator profile metric cards, US-048 lister dashboard stats) need different densities.
