- **Verify RLS via a transactional `begin; ... rollback;` block:** seed fresh fixtures with known `auth.uid`s, then `set local role authenticated` + `set local "request.jwt.claims" = '{"sub":"<uuid>","role":"creator"}'` and assert row visibility via `insert into _asserts ...` rows that you `select *` at the end before rolling back. Important: `grant insert, select on _asserts to authenticated` before switching roles, else you'll get `permission denied for table _asserts`. `PERFORM` is PL/pgSQL only ŌĆö don't use it outside a DO block. - **Trigger function hardening pattern (mandatory for every new trigger / PL/pgSQL function in this repo):**
  1. Declare `set search_path = ''` on the function (CREATE OR REPLACE drops prior SET clauses, so restate it on every redefinition).
  2. With empty `search_path`, qualify EVERY user-defined ref: tables `public.<tbl>`, types/enum casts `public.<type>` (e.g. `'tiktok'::public.platform`), `app_private.<fn>` for sibling functions.
  3. For pg_catalog **functions** (`now()`, `hashtextextended()`, `pg_advisory_xact_lock()`, etc.): bare is fine ŌĆö `pg_catalog` is always implicitly searched even with empty `search_path`. Schema-qualifying as `pg_catalog.<fn>()` also works for true functions.
  4. **GREATEST / LEAST / COALESCE / NULLIF are SQL parser constructs, NOT functions.** They CANNOT be schema-qualified (`pg_catalog.greatest()` raises 42883). Always write them bare.
  5. SECURITY DEFINER functions especially require empty `search_path` ŌĆö without it, a caller-controlled search_path can shadow `public.<table>` and divert writes. The advisor `function_search_path_mutable` flags this at WARN.

- **JWT helper contract (`supabase/functions/_shared/jwt.ts`):** `signJwt({sub, app_role, session_id}, {ttlSeconds?})` + `verifyJwt(token): JwtClaims`. HS256 via Web Crypto. - **Public async wrappers must be declared `async` if they do any synchronous validation that might throw**, so the throw surfaces as a Promise rejection rather than a synchronous exception. `assertRejects` in `@std/assert` fails with "Function throws when expected to reject" if the function throws synchronously ŌĆö a real footgun that can only be caught by actually running the test.
- **Constant-time dual-secret compare for rotation-aware webhook auth:** edge functions that authenticate third-party webhooks via a shared secret must (a) compare in constant time (XOR-accumulate over `max(len(a), len(b))` bytes, iterate the longer one), (b) check BOTH `<SECRET>` and `<SECRET>_PREVIOUS` env vars without short-circuit, and (c) return false if neither env var is set. The second point is subtle: `return timingSafeEqual(a, current) || timingSafeEqual(a, previous)` short-circuits when the first matches, which means an attacker who has recovered `current` can time-distinguish whether `previous` is configured. Correct shape is `let matchA = current ? timingSafeEqual(provided, current) : false; let matchB = previous ? timingSafeEqual(provided, previous) : false; return matchA || matchB` ŌĆö both evaluated unconditionally. Reference: `supabase/functions/apify-webhook/index.ts` `verifyWebhookSecret`.

- **Concurrency via version-id pin + FOR UPDATE re-read, not engine duplication in PL/pgSQL.** When an edge function needs to "re-evaluate eligibility against the NOW-current listing version" (spec ┬¦5.5) without duplicating the TS eligibility engine in SQL, the sanctioned split is: (1) edge fn reads `listings.current_version_id` and runs TS eligibility against it, then (2) calls an RPC that re-reads the same listing under `select ... for update`, compares the locked `current_version_id` to `p_expected_version_id`, and returns `LISTING_VERSION_CHANGED` on mismatch so the client re-opens the detail view. The row lock serializes this flow against `app_private.bump_listing_version` (which runs under its own row lock on the listings UPDATE). Invariant depends on the bump trigger producing a new `current_version_id` for every eligibility-affecting mutation ŌĆö if a future trigger change lets a condition edit land without bumping, the RPC would silently commit against stale TS eligibility. Document the dependency in the RPC header. Ref: `supabase/migrations/20260417270000_us_041_apply_to_listing_rpc.sql` + `apply-to-listing/index.ts`.
- **Fail-closed applies to malformed data, not just missing data.** Spec ┬¦3d's fail-closed rule covers the "null actual metric" case (user hasn't been scraped), but the same principle extends to unreadable THRESHOLD values: a `gte` pre-condition with a NULL or non-numeric `numeric_threshold` must fail the check, not silently pass. Silently returning `null` from the single-condition evaluator for a malformed row is how a future mis-migrated condition flips a listing to "always eligible" without any alarm. Rule: every branch of an eligibility-or-permission evaluator that hits unreadable input returns a fail, never an early "skip." Schema CHECKs should prevent the malformed row from existing, but the evaluator is the last line of defense. Ref: `_shared/eligibility.ts` `evaluatePreCondition`.
- **Extract shared modules under `supabase/functions/_shared/` as soon as a second function needs the same logic.** US-039's eligibility engine was inline; US-041 needed the same engine for its server-side re-check. Pattern: extract to `_shared/<name>.ts` with a header comment naming the consumers and the spec sections it implements, have BOTH functions import it via `../_shared/<name>.ts`, and **redeploy every consumer function** in the same iteration ŌĆö the deployed bundle of a prior function still contains the inlined copy, so a future redeploy of that function without the shared file would fail with `Module not found`. The `deploy_edge_function` `files` array must include every `_shared/*.ts` dependency, not just the new one. Ref: `_shared/eligibility.ts` shared by `get-listing-detail` (v4) + `apply-to-listing` (v2).
- **`KeyboardAvoidingView` is unreliable inside an RN `<Modal>` (and therefore inside our `BottomSheet`).** The Modal opens its own native window, so KAV's measured frame and the keyboard's reported frame disagree and the contents do not get pushed up. Replace KAV with a direct `Keyboard.addListener('keyboardWillShow' / 'keyboardWillHide')` (use `keyboardDidShow`/`keyboardDidHide` on Android) that drives `paddingBottom` on the inner `<View>` from `e.endCoordinates.height`. Remember that `BottomSheet` snap point must accommodate `keyboard_height (~336px iOS) + content_height` ŌĆö for the apply sheet that meant raising `snapPoints` from `[0.7]` to `[0.85]`. Ref: `src/screens/listing-detail/ApplySheet.tsx`.
- **Defer `router.replace`/`router.push` after dismissing a `BottomSheet` by ~350ms** so the sheet's spring-out animation (`SPRING_SNAPPY` settles in ~300ms ŌĆö see `src/design/motion.ts`) plays cleanly before the new screen mounts. Without the delay, the sheet slides down on top of the freshly-rendered destination, which looks broken. The same delay applies to any imperative navigation triggered from a Modal/BottomSheet success path. Ref: `app/(creator)/listing/[id].tsx` `onApplied`.
- **Run cheap checks before upstream oEmbed/scraper calls in edge functions; return `502 OEMBED_UNAVAILABLE` (not 500) when the upstream itself fails.** Order inside an endpoint that relies on a third-party URL validator: JWT ŌåÆ role ŌåÆ parse body ŌåÆ regex-classify URL ŌåÆ DB lookups ŌåÆ status/permission gates ŌåÆ affirmation/eligibility check ŌåÆ upstream oEmbed ŌåÆ RPC. This is both a cost optimization (don't burn TikTok rate-limit slots on requests that will 4xx regardless) and a DDoS-amplifier guard (an unauth POST must not be able to force an upstream fetch). Split the error codes: `500 SERVER_MISCONFIGURED` / `500 DB_ERROR` for our-fault failures, `502 OEMBED_UNAVAILABLE` for upstream unavailability ŌĆö clients that auto-retry benefit because upstream outages are usually retryable, DB bugs rarely are. Document BOTH codes in the contract comment. Ref: `supabase/functions/submit-video/index.ts` oEmbed section + error envelope header.
- **Defensive null-guards on NOT NULL columns fetched via service-role client, when the row feeds a security-relevant gate.** The database schema can enforce NOT NULL, but a TypeScript read from a future mis-migrated row that returns `null` will silently propagate into downstream logic. When the value gates a security check (e.g. `application.listing_version_id` ŌåÆ the post-condition affirmation check; an empty `.eq('listing_version_id', null)` fetch returns zero rows ŌåÆ no affirmations required ŌåÆ gate bypassed), add an explicit `if (!row.field) { 500 DB_ERROR }`. One extra branch is cheaper than a future exploitable silent-pass. Ref: `supabase/functions/submit-video/index.ts` listing_version_id guard.
- **PostgREST embeds silently NULL joined columns when the target table's RLS filters the caller out ŌĆö always audit every `.select('..., related!fk(col)')` embed against the related table's RLS policies.** Failure mode: the embed syntax succeeds, returns rows, and populates the parent columns; the embedded field just comes back as `null` or `[]` under RLS because PostgREST applies the policies per-row transparently. Highest-risk embeds on this project: (a) `users!lister_id(username)` where `users_self_select` is self-only ŌĆö every cross-tenant handle read (creator viewing lister handle, lister viewing applicant handle) will null out; (b) `listing_versions!...(...)` where `listing_versions_read_if_listing_readable` gates on `listings.status='active' OR caller=lister` ŌĆö any historical/closed listing version the caller doesn't own will null; (c) `listings!...(title)` where `listings_public_read` requires `status='active'` ŌĆö same closed-listing nullout. Rule: before writing a PostgREST embed, check the target table's RLS; if the policy wouldn't return the specific row under the caller's role, replace the embed with a SECURITY DEFINER RPC that does its own ownership check. DO NOT widen the self-only policy to "return usernames if handle is public" ŌĆö that leaks the full row shape; DEFINER + narrow column projection is the correct primitive. Ref: Codebase Pattern #40 (`submission_reuse_count`), Codebase Pattern #110 (`apply_to_listing_rpc`), and `public.list_my_applications()` from US-043.
- **SECURITY DEFINER "list-my-rows-with-foreign-context" RPC skeleton.** Contract: `returns table(<flat columns including the cross-tenant bits>)`, `language sql stable security definer set search_path = ''`, body is a straight `select ... from public.<owned_table> t left join public.<foreign_table> f on ... where t.<owner_col> = public.current_user_id() order by ...`. Grants: `revoke all ... from public`, `revoke all ... from anon`, `grant execute ... to authenticated`. Service-role is intentionally not granted ŌĆö it already has BYPASSRLS and wouldn't use this shape. Why `stable`, not `volatile`: the function reads but doesn't write, so PostgREST can plan around it and query planner caches can reuse results within a statement. Why `sql` over `plpgsql`: the body is one statement, no control flow, no GET STACKED DIAGNOSTICS ŌĆö inlines better. Verified pattern via a transactional `begin; set local role authenticated; set local "request.jwt.claims" = '{"sub":"<uuid>","role":"creator"}'; select * from public.<fn>();` block that confirms (a) the DEFINER bypass returns the cross-tenant columns, (b) the ownership check still gates other users' rows (swap the sub claim to a different creator and assert zero rows). Ref: `supabase/migrations/20260418010100_us_043_list_my_applications_rpc.sql`. Applies to any future "my X with lister/creator/reviewer context" list screen.
- **Screen-RPC naming convention: `public.list_my_<things>` for caller-scoped list reads, `public.<action>_rpc` for mutations, `public.<thing>_<metric>(arg)` for aggregates.** Precedents: `public.list_my_applications()` (US-043) / `public.apply_to_listing_rpc(...)` (US-041) / `public.submission_reuse_count(uuid)` (US-009). Keep the `list_my_*` functions argumentless (the caller is implicit via `current_user_id()`) ŌĆö passing a caller UUID would re-introduce the spoofing surface the helpers exist to eliminate. Keep them in the `public` schema (not `app_private`) because they need to be callable via PostgREST `supabase.rpc(...)`; `app_private` is reserved for server-side triggers + cron bodies. The `authenticated` grant is mandatory ŌĆö without it PostgREST returns 42501 "permission denied for function" and the MCP `execute_sql` test path (runs as `postgres`) silently passes, masking the bug.
- **Segmented control a11y: `accessibilityRole="tablist"` wrapper + `accessibilityRole="tab"` on each button + `accessibilityState={{ selected }}`**, NOT the Chip primitive's `role="checkbox"` + `checked`. Chip is for multi-select filter rows where each pill is independent; a segmented control is single-select and semantically is a tab bar. iOS VoiceOver reads "tab, Pending, selected, 1 of 4" for the tablist shape ŌĆö correct under Apple's a11y guidance. `selected` is the right state (not `checked`) because the user isn't toggling a boolean, they're picking one of N mutually-exclusive views. Inline the control rather than promoting to a shared primitive until a second screen needs it ŌĆö premature abstraction hurts more than a ~40-line duplicate. Ref: `SegmentControl` in `app/(creator)/applications.tsx` (US-043).
